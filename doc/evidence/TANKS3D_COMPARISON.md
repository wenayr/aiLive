# Tanks 3D: первый сопоставимый прогон

## Вопрос

Что фактически меняется, если небольшую браузерную игру развивать напрямую или
как product target внешней development-песочницы с генераторами?

## Устройство двух вариантов

| Вариант | Первая итерация | Вторая итерация |
| --- | --- | --- |
| `projects/tanks3d-simple/` | Один `game.js`: ручная арена, ручные танки и renderer | Три вручную заданные reinforcement squads в том же runtime |
| `projects/tanks3d-sandbox/` | Отдельные arena, tank и wave generators, game runtime и renderer | `round` стал единственным входом роста; runtime просит следующую волну у wave generator |

Оба варианта — локальные статические браузерные игры с изометрическим Canvas
рендером, движением, башней, выстрелами и противниками. Они не импортируют код
друг друга.

## Третья итерация: карты, модели и точный прицел

Один и тот же новый запрос был применён к обоим продуктам:

- три выбираемых вида карты: `crossroads`, `ring`, `islands`;
- видимые параметры танка: корпус, гусеницы, башня, ствол и скорость снаряда;
- прицел игрока в мировых координатах от позиции мыши;
- снаряд прекращает движение при столкновении со стеной.

| Вектор требования | Simple | Sandbox |
| --- | --- | --- |
| Карты | Три ручных массива `manualMaps` в `game.js` | `createArena({kind})` и три profile-функции с детерминированным входом |
| Модели | Условный выбор model внутри `tank()` | Archetype contract в `createTank()`, который получает renderer и physical shell path |
| Прицел и физика | Pointer → inverse projection → монолитный runtime | Pointer → `input.aim` → `createGame`; model задаёт скорость shell |
| Проверка | Один smoke-test структуры монолита | Generator properties, map kinds, model dimensions, runtime aim и wave transition |

### Наблюдаемая цена кода

В этой конкретной итерации чистый прирост игрового кода составил примерно
`+32` строки в simple runtime и `+58` строк в sandbox product sources. В
sandbox затронуты семь продуктовых поверхностей: arena/tank/wave generators,
runtime, renderer, browser binding и tests; в simple — главный runtime и
страница выбора карты. Это не измерение времени человека, но честный индикатор
первоначальной архитектурной цены: в данном запросе sandbox примерно в `1.8×`
тяжелее по чистому объёму изменения.

### Вектор трудозатрат дальше

| Следующий тип работы | Simple | Sandbox после этой итерации |
| --- | --- | --- |
| Четвёртая карта | Новый ручной массив, плюс проверка, что он не ломает движение | Одна profile-функция и детерминированный test; runtime и renderer не меняются |
| Новый танк | Новая ветка характеристик, ручная настройка рендера и выстрела | Новый archetype record; renderer уже читает model contract |
| Новый режим волны | Ещё один список/ветка в одном runtime | Изменение `createWave`; existing instruction попросит проверить сохранение source of truth |

Следовательно, на одном или двух вариантах simple почти наверняка дешевле. Точка
проверки sandbox — не этот первый рост, а четвёртая карта, новый archetype и
вторая похожая правка wave generator: там нужно сравнить время, повторные ошибки
и число затронутых мест, а не только строки кода.

## Зафиксированный sandbox-цикл

1. После первой итерации `development/tanks3d-sandbox/` выполнил `setup` и
   сохранил baseline четырёх tracked product paths.
2. Во второй итерации изменены только `create-game.mjs` и `create-wave.mjs`.
3. `scan` создал один `owner/discover` batch с этими двумя путями.
4. Владелец выбрал `discover/skip`: одно корректное изменение ещё не доказывает
   повторяемую ошибку или необходимость инструкции.
5. После `resolve`: `batchCount: 1`, `observationCount: 0`, pending отсутствует.
6. Третья итерация изменила arena/tank/wave generators и runtime. Это второй
   реальный повтор одной границы, поэтому `discover/record` сохранил instruction:
   менять generator contracts без копирования map/model/spawn data в runtime и
   подтверждать generator property вместе с затронутым runtime behavior.
7. После второго `resolve`: `batchCount: 2`, `observationCount: 1`,
   `activeInstructions: 1`, pending отсутствует.

## Что этот прогон показывает

- Внешняя песочница действительно наблюдает сохранённый продукт, не являясь его
  внутренним модулем и не изменяя его во время `setup`.
- Генеративная граница проверяема: один round создаёт следующий набор танков из
  spawn pads, а не из скопированного в runtime списка.
- Simple-вариант быстрее и прямее для ручного контента; sandbox-вариант требует
  дополнительной архитектуры, зато следующая волна растёт через один источник
  истины и имеет unit-проверку.

## Чего этот прогон не показывает

Он не измеряет скорость человека, токены, повторные ошибки или экономический
эффект на серии задач. Для этого нужна минимум ещё одна похожая продуктовая
правка и сопоставимый baseline без sandbox. Тогда можно измерять время, число
ремонтов, ложные срабатывания и полезность конкретной инструкции.
