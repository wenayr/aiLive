# Первый поток: итог genetic module experiment

Дата: 2026-07-15.

## Что подтвердилось

Малый внешний genetic loop работает как архитектурная граница: он хранит
состояние отдельно от продукта, реагирует на сохранённые наблюдаемые файлы до
Git commit и не требует встроенного provider API. В generator-варианте
повторяемый ресурс остаётся компактным source of truth и не смешивается с
browser runtime.

В сравнении был заметен ограниченный плюс: generator-вариант легче удерживал
согласованную структуру повторяемого ресурса при изменениях, тогда как прямой
вариант быстрее собирал один локальный экран. Это ещё не доказательство общей
выгоды по времени, токенам или качеству.

## Что исказило эксперимент

- обе ветки находились в одном большом разговорном контексте;
- управляющий агент несколько раз навязал конкретные решения обеим веткам;
- ранние попытки смешали исторические карты, новые сцены и несколько режимов;
- точные provider token usage не были доступны;
- browser проверял первый кадр, но недостаточно долго проверял живую сцену.

## Правила второго потока

1. Одна задача — один короткий intent brief без списка решений.
2. Каждая ветка получает независимый агентный контекст и отдельный usage log.
3. До начала фиксируются время, provider tokens, ремонты, product files и
   визуальные критерии.
4. Генератор появляется только после наблюдаемого повторения.
5. Внешний genetic module вызывается после сохранённой правки, а результат
   `discover/skip/inspect` сохраняется как evidence.
6. Browser smoke ждёт стабильный runtime-отрезок, проверяет HUD и console.
7. После потока остаётся один итоговый отчёт и один выбранный fixture; всё
   промежуточное удаляется либо остаётся только в Git history.
