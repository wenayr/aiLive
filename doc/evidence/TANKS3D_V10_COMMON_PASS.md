# Tanks 3D: общий V10-проход

Дата: 2026-07-15.

## Вопрос

Можно ли передать чистой версии и версии с внешней development-песочницей один
абстрактный запрос «сделай игру классной» так, чтобы сравнивать не случайные
разные фичи, а одинаковую пользовательскую поверхность?

## Метод

Обе версии получили один и тот же обязательный результат из десяти независимых
векторов:

1. визуальная личность карты;
2. тактический ресурс на карте;
3. превращение ресурса в мобильность игрока;
4. различимые роли противников;
5. обратная связь попадания;
6. счёт за уничтожение;
7. нарастание волн;
8. пауза и рестарт;
9. видимое управление;
10. воспроизводимая проверка карт и правил.

Точный brief: [simple](../../projects/tanks3d-simple/BRIEF_V10.md) и
[sandbox](../../development/tanks3d-sandbox/BRIEF_V10.md). Разрешалось выбрать
разную внутреннюю раскладку, но не менять пользовательский контракт.

## Единственный проход

| Вектор | Simple | Sandbox product |
| --- | --- | --- |
| Карты и палитры | Ручные конфигурации карт в одном runtime-файле | `createArena` возвращает палитру и core pads |
| Ресурс → turbo | Runtime хранит cores и charges | Runtime материализует generator-owned core pads |
| Роли и награда | Ручные archetype/bounty | `createTank` возвращает archetype и bounty |
| Попадание, счёт, волны | Runtime effects, score и ручная волна | Runtime effects/score поверх generator output |
| Пауза, рестарт, guide | В верхнем уровне приложения | В верхнем уровне приложения |
| Воспроизводимость | `manual-content.test.mjs` | generator/runtime/renderer tests |

То есть сравнение не навязывает simple-версии генераторы, а sandbox-версии —
ручные данные. Оно требует одинаковый наблюдаемый продукт.

## Проверка

Выполнены команды:

```text
npm --prefix projects/tanks3d-simple run verify
npm --prefix projects/tanks3d-sandbox run verify
```

Результат: simple — 1/1 test; sandbox — 9/9 tests. Отдельный browser-smoke
для `?map=ring` показал в обеих версиях выбранную Ring-карту, canvas, HUD с
`score`, `cores`, `boost/turbo`, картой и волной, а также guide управления.
В свежих вкладках не было browser console errors или warnings.

## Что показала песочница

Внешний генетический модуль увидел сохранённые изменения трёх tracked-файлов:
`create-arena`, `create-tank`, `create-game`. Его уже активная инструкция
проверила границу: карта, модель и spawn-правила остаются в генераторах, а
runtime только потребляет их возвращаемые данные. Инспекция завершилась
`clear`; новая инструкция не создавалась, pending action отсутствует.

Это не означает, что песочница проверила всю игру. Первый browser-smoke поймал
две ошибки рендерера: helper-ы обращались к несуществующим внешним `arena` и
`player`. Рендерер не входил в tracked paths, поэтому генетическая инспекция
их и не должна была увидеть. Ошибки исправлены передачей явных зависимостей в
renderer и покрыты новым `renderer.test.mjs`.

## Вывод

Один общий абстрактный brief можно исполнить сопоставимо: внешний результат у
двух вариантов одинакового класса, а внутреннее решение остаётся свободным.
Эта итерация не доказывает, что sandbox уже быстрее: она показала его текущую
цену и пользу. Он удержал генераторный контракт в изменении, но его scope надо
явно расширять только тогда, когда повторится класс проблем в рендеринге; не
следует превращать единичную ошибку в новый «генетический» слой.
