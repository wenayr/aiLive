# Genetic baseline

## Уже подтверждено реализацией и тестами

- baseline и последующие ревизии файлов можно сравнивать по содержимому без Git
  commit;
- создание, изменение и удаление файла представимы одной change-моделью;
- повторное одинаковое событие может быть подавлено event gate;
- при изменении выбираются только инструкции, чьи watched paths пересекаются с
  изменёнными файлами;
- следующий scan блокируется, пока не разрешено текущее действие;
- после discovery инструкция участвует в следующей связанной правке;
- clear-inspection завершает цепочку без лишнего действия, а issue требует
  отдельного решения владельца `keep/refine/retire`;
- слабое discovery можно завершить без создания наблюдения и инструкции.

Эти свойства проверялись ранними core-тестами и сохранены в текущем
самостоятельном проекте. Старый provider/fixture harness удалён: он доказывал
работу контроллера симуляции, но не эффективность генетической идеи.

## Текущий эксперимент

15 июля 2026 года самостоятельная песочница прошла живой контролируемый цикл.
Исполнителем был Codex в каталоге проекта; fixture, provider adapter и внешний
model API не использовались.

Последовательность:

1. `npm run setup` создал baseline одного файла без Git commit.
2. Добавление Bybit и `npm run scan` создали `owner/discover`.
3. Было сохранено наблюдение `exchange-adapter-canonical-normalization` и
   инструкция проверять делегирование в `normalizeQuote` с именем источника.
4. Отдельный CLI-процесс восстановил state. Добавление OKX и следующий scan
   выбрали эту инструкцию и создали `reviewer/inspect` только для изменённого
   `src/exchanges.mjs`.
5. Инспекция дала `clear` и завершила цепочку без дополнительного owner action.
6. `npm run check:bybit`, `npm run check:okx` и все 11 tests через
   `npm run verify` прошли; чистый корневой `npm ci` также подготовил workspace.
7. Полный каталог был скопирован вне родительского репозитория; независимые
   `npm ci`, `npm run verify`, `npm run setup` и `npm run check:baseline` прошли.

Финальное локальное состояние: 2 change batches, 1 observation, 1 active
instruction, 1 inspection, `approvedChecks: 1`, 0 pending actions. Два принятых
ответа сохранены в history, порядок setup/scan/resolve — в trace. Каждая команда
CLI запускалась отдельным процессом, поэтому цикл также подтвердил дисковое
восстановление состояния. Повторный scan без правок не создал третий batch.

Это доказывает рабочую механику, границы каталогов и перенос инструкции на
следующую связанную правку. Оно не доказывает причинное улучшение качества:
OKX сразу был написан правильно, а сопоставимого baseline без инструкции не
было.

## Пока не доказано

- что инструкция снижает ошибки по сравнению с тем же агентом без неё;
- что экономия превышает накладные расходы на реальном проекте в несколько
  тысяч строк;
- что автоматический post-edit hook лучше явного scan;
- когда наблюдение стоит превращать в guard, генератор или отдельный MCP;
- как переносить полезную практику между проектами без загрязнения контекста.

Для оценки эффективности нужен повторяемый A/B-набор задач с одинаковой моделью
и контекстом: baseline без инструкции против варианта с выбранной инструкцией,
с измерением ошибок, ремонтов, времени, токенов и ложных срабатываний.
