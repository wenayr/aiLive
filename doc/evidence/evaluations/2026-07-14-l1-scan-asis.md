## Public surfaces and consumers actually present

- Core functions: scenario construction/validation, initial state, fixed step, hash, replay.
- Coordinator runtime: `snapshot`, `submitCommand`; debug: replayed updates, metrics, event history; testing: manual advance and queue observation.
- RPC facade deliberately omits the coordinator testing surface.
- Consumers observed: headless two-bot runner, local authoritative service, Arena browser client, and the L0 host app. The host app and Arena client are separate client factories, but only one dedicated Arena browser view is present.

## Verification map

- Nine L1 tests cover seeded connectivity, canonical command ordering, blocked movement, destruction, live/replay equality, facade separation, protocol incompatibility, headless bots and coordinator update emission.
- `npm run acceptance:realtime` records a deterministic trace (`badf946a` live and replay hash).
- The service/UI smoke confirms a live local loop and browser subscription.

## Tensions and documentation contradictions

1. `BUILD_PLAN.md` asks for two Arena web clients, tick lag, desync count, debug timeline/state inspector and test-only event injection. The current code has one Arena panel, current tick/tick interval/state hash/last events and queue-fire controls. It has none of the listed lag/desync/timeline/injection surfaces.
2. `layer.yaml` lists debug `inspect state` and testing `inject command`, `replay trace`, `hash state` at the layer level. The core simulation exposes `inject` and command trace, while the coordinator testing facade instead exposes `advance` and queued commands; the RPC facade exposes neither testing API. The manifest is therefore an aggregate intent, not an exact facade inventory.
3. Incoming RPC commands are statically typed in TypeScript and protocol-version mismatch is tested in the simulation, but this scan found no runtime structural codec/validator at the Socket.IO boundary. Malformed remote payload behaviour is unverified.
4. Browser reconnect code resubscribes and refreshes after hub reconnection, but no test or retained trace simulates service restart, missed update sequence or a desynchronised mirror.

## Unknowns

- Whether a second independent Arena browser consumer is genuinely useful, rather than a test harness being the correct second consumer.
- Whether protocol-boundary validation, sequence recovery or a timeline inspector recur enough to justify a new boundary instead of local completion work.
- Whether one local session is the intended product shape; service construction currently hard-codes one scenario/session inside the L0 binding.
- Actual latency, packet-loss behaviour, concurrent-client semantics, process-restart recovery and client desynchronisation.

## Scan conclusion

No existing folder is promoted. The evidence supports keeping the current Arena code local while a later design pass decides whether to first close the documented operational gaps or evaluate a bounded candidate. Any growth decision must begin with a repeated change pressure and candidate-scoped experiment; this scan does not supply either.
