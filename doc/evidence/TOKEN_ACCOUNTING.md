# Token accounting: честная база для сравнений

Дата: 2026-07-15.

## Что нельзя восстановить задним числом

V11 simple и sandbox были сделаны внутри одного общего Codex-диалога. В Git
есть финальные файлы и diff, но нет provider usage: prompt/input tokens,
completion/output tokens, cached tokens, reasoning tokens и число повторных
генераций. Поэтому нельзя честно сказать «simple потратил N токенов, sandbox
потратил M» для уже завершённого V11. Размер кода не является этим числом.

Sandbox в этом срезе создал на 19% больше добавленного product/test текста и

По финальному acceptance оба варианта закрыли один и тот же V11 contract:
один танк, мышь для башни, W/S/A/D, стрельба, разрушаемые блоки, десять карт и
видимый HUD. На этом одном прогоне нельзя заявить радикальный выигрыш качества:
результат принят в обеих ветках; sandbox дал более сильную локальную проверку
(12 product tests против 2), но это также часть его цены.

## Что считать в следующем запуске

Каждый отдельный агентский/model вызов должен оставить одну запись в JSONL:

```json
{
  "runId": "v12-sandbox",
  "variant": "sandbox",
  "step": "implement-field-extension",
  "provider": "provider-name",
  "model": "model-name",
  "inputTokens": 0,
  "cachedInputTokens": 0,
  "outputTokens": 0,
  "reasoningTokens": 0,
  "accepted": true,
  "filesChanged": ["relative/path"],
  "verification": {"command": "npm run verify", "passed": true}
}
```

Числа берутся только из `usage` ответа конкретного провайдера или агентского
раннера, а не из оценки длины текста. Если поле не выдаётся провайдером, в нём
остаётся `null`, а не выдуманный ноль.

## Итоговые формулы

Для каждой ветки и задачи считать отдельно:

```text
input       = Σ inputTokens
cache       = Σ cachedInputTokens
output      = Σ outputTokens
reasoning   = Σ reasoningTokens
generated   = output + reasoning
total_seen  = input + cache + output + reasoning
accepted_output = generated только у шагов, вошедших в принятый результат
```

`cache` нельзя молча складывать с платным input: его тариф зависит от
провайдера. Денежную стоимость считать отдельной функцией от токенов и
конкретного прайса на дату запуска.

## Эффективность

Сравнивать не один коэффициент, а три оси:

1. **Цена:** `generated`, `total_seen` и при наличии прайса стоимость.
2. **Результат:** доля обязательных требований, прошедших tests/browser-smoke.
3. **Надёжность:** число product defects до финального acceptance и число
   повторных правок той же причины.

Тогда sandbox может проигрывать по `generated`, но выигрывать по ошибкам или
стоимости следующей похожей задачи. Это и будет содержательным выигрышем, а не
красивым числом строк.
