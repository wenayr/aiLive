# Контракт ответственной песочницы

## Два взгляда на один каталог

Родительский репозиторий видит `sandbox/genetic-module/` как внутренний каталог.
Если открыть этот каталог отдельным проектом Codex, он становится полным рабочим
корнем и не зависит от файлов родителя. Вложенный `.git` для такого эффекта не
нужен.

Внутри находятся две живые области:

```text
project/   код, который агент меняет для текущей задачи
genetic/   состояние, pending action, ответы, история и трасса модуля
```

Обе области локальны и игнорируются Git. В Git остаются реализация модуля,
шаблон проекта, задачи, тесты и отобранное evidence.

## Ответственность

- Codex читает задачу и меняет только `project/`.
- Генетический runtime читает только настроенные tracked paths из `project/` и
  пишет служебные данные только в `genetic/`.
- Codex может заполнить `genetic/response.json` по выданному шаблону, но не
  правит `state.json`, `pending.json`, `history/` и trace вручную.
- Один Codex может последовательно сыграть роли owner и reviewer. Отдельный
  агент повышает независимость оценки, но не требуется для проверки механики.
- Ни runtime, ни CLI не вызывают модель, сеть, Git, браузер или внешний MCP.

## Рабочий протокол

1. `npm install` и `npm run verify` внутри самостоятельного каталога.
2. `npm run setup` один раз создаёт `project/` и `genetic/`; существующие
   данные не перезаписываются.
3. Агент читает задачу, меняет файл и сохраняет его.
4. `npm run scan` сравнивает диск с сохранённым baseline.
5. Если создан `genetic/pending.json`, агент выбирает основной или альтернативный
   response template, сохраняет его в `genetic/response.json`, заменяет
   шаблонный текст конкретным заключением и запускает `npm run resolve`.
6. Если resolve создал следующее действие, шаг 5 повторяется до состояния ready.
7. Явная предметная проверка задачи проверяет код; `npm run status` показывает
   состояние.

Событие первой версии — явный scan сразу после сохранения файла. Позднее его
можно привязать к post-edit hook или file watcher с debounce. Polling каждой
реплики и Git push не нужны. Git diff может стать дополнительным источником
контекста, но не основным моментом срабатывания.

## Надёжность первой версии

State валидируется при восстановлении, пишется через временный файл и rename,
а параллельные команды блокируются lock-файлом. Пути нормализуются относительно
`project/`, tracked symlink отклоняется, существующая сессия не сбрасывается
молча. Это защищает эксперимент от случайной порчи, но не превращает каталог в
безопасную среду для исполнения недоверенного кода.
