# Текущая модель живых модулей

Статус: рабочая исследовательская модель. Она описывает направление экспериментов, а не уже доказанную универсальную архитектуру.

## 1. Главная гипотеза

Обычный проект можно строить привычным способом, но оставлять в нём хорошо различимые ресурсные, преобразующие, горизонтальные и фасадные слои. По мере использования вокруг некоторых слоёв накапливается генетический ореол:

```text
код и типы
→ локальные проверки
→ архитектурное описание
→ примеры и usage profiles
→ materialization routes и generators
→ самостоятельный контракт
→ package / service / MCP, если это оправдано
```

Генетический модуль наблюдает этот процесс, восстанавливает фактические границы и предлагает проверяемые варианты развития. Он не обязан превращать каждый участок в модуль. `KEEP_LOCAL` и `NO_GENERATOR` являются полноценными результатами.

## 2. Что именно развивается

Нужно различать три независимые оси:

1. Структурная роль: resource, transform, horizontal, facade, generator, verifier, coordinator или composite.
2. Зрелость: от неявного fragment до доказанного или развивающегося модуля.
3. Физическая форма: каталог, workspace package, versioned artifact, repository, runtime или MCP.

MCP — транспортная проекция для агентного доступа, а не высшая степень зрелости.

## 3. Генераторизация

Генераторизация — отдельная гипотеза развития участка. Код может остаться источником истины, а generator создавать только mocks, tests или adapters. Маленький слой может иметь большое генеративное плечо; большой слой может вообще не нуждаться в generator.

Для каждого участка независимо рассматриваются:

- основной код;
- schemas, types и codecs;
- mocks, fixtures и scenarios;
- tests и test matrix;
- adapters и project bindings;
- consumer/debug/testing facades;
- конфигурации и документация;
- архитектурный scaffold;
- MCP-проекция;
- композиционный маршрут других модулей.

До реализации generator candidate обязан объявить семейство выходов, variation dimensions, source of truth, ownership mode, проверку, fallback и ожидаемую стоимость сопровождения.

## 4. Многоуровневые фасады

Одна каноническая операция может присутствовать в нескольких фасадах. Фасад не владеет копией логики: он задаёт проекцию для потребителя, уровень абстракции и права.

Пример:

```text
transport.send
├── runtime facade
├── debug facade
└── testing facade
```

Debug-потребителю может быть доступна более широкая поверхность, чем приложению. Это даёт локальность расширений без искусственного правила «один метод — один потребитель».

## 5. Core, Profile, Binding

Для кандидатов этой системы принимается рабочее разделение:

- `Module Core` — самостоятельная способность и её инварианты;
- `Usage Profile` — доказанный типовой вектор применения и defaults;
- `Project Binding` — связь с auth, state, runtime, deployment и другими решениями конкретного проекта.

Это архитектурное решение проекта живых модулей, а не универсальная онтология, которую SCAN обязан обнаружить в любом чужом репозитории.

## 6. Генетический модуль и генетическая база

- `Генетический модуль` — наблюдатель и проектировщик candidate-изменений.
- `Генетическая база` — накопленные карты, решения, profiles, traces, rejected candidates и evidence, которыми он пользуется.

В будущем модуль может оказаться композицией scanner, boundary designer, generator hunter, package foundry и reviewer. Пока не следует преждевременно фиксировать его внутреннее устройство.

## 7. Четыре плоскости документации

1. Исследовательский диалог и design pressures сохраняют замысел и альтернативы.
2. Спецификация фиксирует маленькое текущее нормативное ядро.
3. Операционные инструкции управляют одним режимом и считаются заменяемыми.
4. Evidence и решения определяют, какая гипотеза доказана и чему разрешено стать нормой.

Prompt не является механизмом безопасности и не может сам подтвердить свою эффективность.

## 8. Рабочий цикл

```text
BUILD реального baseline
→ SCAN фактического AS-IS
→ интерпретация напряжений
→ GENERATOR_HUNT / boundary alternatives
→ ACQUIRE внешних вариантов при необходимости
→ candidate в изолированной области
→ локальная и проектная проверка
→ независимый REVIEW
→ PROMOTE / PROFILE / INLINE / ARCHIVE / REJECT
→ обновление genetic base
```

Observation отделяется от design, а design — от materialization и promotion. Это защищает систему от поиска только тех сущностей, которые заранее перечислены в инструкции.

## 9. Таксономия действий

Действия разных классов нельзя смешивать в один список.

Наблюдение и обогащение:

- `ANNOTATE`, `PROFILE`, `PROTECT`, `PROPOSE_GENERATOR`.

Материализация candidate:

- `GENERATE`, `EXTRACT`, `FUSE`, `EXPOSE_API`, `EXPOSE_MCP`.

Жизненный цикл:

- `KEEP_LOCAL`, `PROMOTE_CANDIDATE`, `PROMOTE_STABLE`, `INLINE`, `ARCHIVE`, `REJECT`, `PRUNE`.

Первые эксперименты не обязаны поддерживать все действия.

## 10. Критерий полезности

Главная проверка: стала ли следующая связанная задача дешевле и надёжнее при контролируемой стоимости интеграции и сопровождения.

Для этого нужны baseline и горизонт измерения. Дополнительные показатели:

- Context Reduction;
- Independent Verification;
- Glue Ratio;
- Transfer Rate;
- Regression Rate;
- Generator Maintenance Cost;
- False Extraction Rate;
- External Adoption Yield.

Количество модулей, generators, MCP и строк instructions успехом не считается.

## 11. Что пока неизвестно

Не доказаны универсальность онтологии слоёв, качество автоматического поиска границ, обязательность второго потребителя, числовые пороги scorecard, безопасность self-evolution, выгодность macro-module synthesis и способность одной инструкции работать на разных доменах.

Поэтому начинать нужно с нескольких сложных реальных участков и фиксировать false positives, разногласия reviewers и стоимость отката.
