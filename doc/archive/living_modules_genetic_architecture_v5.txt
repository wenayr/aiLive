# Живые модули, динамические инструкции и генетическая база развития

## Каноническое объяснение замысла, варианты архитектуры, стратегия запуска и исследовательские промпты

**Версия:** 1.0 / июль 2026  
**Назначение:** перенос контекста в новый чат, архитектурная основа прототипа, документ для критического обсуждения и план первых экспериментов.

---

## 0. Как читать этот документ

Этот текст не является утверждением, что уже найден единственный правильный стандарт. Наоборот, одна из главных мыслей состоит в том, что **идеальная стратегия развития не может быть одной для всех состояний проекта**.

Здесь разделены три типа утверждений:

- **Подтверждённая инженерная основа** — практики, которые уже используются в software engineering, Agent Skills, MCP, eval-системах, каталогах и архитектурном восстановлении.
- **Текущее архитектурное решение** — предлагаемая конструкция, которую уже можно реализовать и проверить.
- **Исследовательская гипотеза** — потенциально сильная часть системы, которую нельзя считать доказанной до серии реальных проектов.

Документ отражает весь ход нашего диалога, включая моменты, где первоначальная трактовка была слишком тяжёлой или слишком централизованной. Поэтому здесь специально указано не только **что предлагается**, но и **какие формулировки были отброшены или уточнены**.

# Часть I. Нить сути

## 1. Идея в одном абзаце

Мы строим не просто набор MCP-серверов и не просто библиотеку промптов. Мы строим среду, в которой обычные проекты, библиотеки и генераторы постепенно обрастают **динамическими инструкциями, многоуровневыми фасадами, локальными проверками, примерами и накопленным опытом**. Часть развивающихся слоёв со временем становится самостоятельными живыми модулями. Зрелый модуль умеет консультировать, генерировать новое, интегрироваться в существующий проект и перестраивать уже написанную реализацию под свою архитектуру. Когда готового модуля нет, проект можно быстро сделать обычным агентом; затем **генетическая база развития** восстанавливает фактическую архитектуру, выделяет полезные способности и превращает их в наследуемые инструкции, генераторы, профили и иногда новые модули. MCP остаётся одним из стандартных интерфейсов доступа, но истинной единицей системы является не MCP-сервер, а **способность, представленная динамической инструкцией, API-фасадом, генератором или иным provider и локальным доказательством работоспособности**.

## 2. Самая короткая формула

```text
Живой модуль =
    самостоятельная способность
  + многоуровневый фасад
  + динамическая инструкция
  + генератор / provider
  + профили использования
  + локальные доказательства
  + история развития
```

```text
Генетическая база =
    восстановление архитектуры
  + добыча способностей из проектов
  + поиск готовых модулей
  + создание недостающего
  + split / fuse / inline / prune
  + candidate-изменения
  + независимая проверка
```

```text
Проект =
    контекст задачи
  + выбранные живые модули
  + локальные слои, которые ещё не стали модулями
  + project bindings
  + артефакты и evidence
```

## 3. Центральный двунаправленный цикл

```text
Прямое направление

зрелый модуль
→ получает контекст проекта
→ выбирает маршрут и provider
→ генерирует / интегрирует / перестраивает
→ проверяет результат
→ возвращает артефакт


Обратное направление

быстро созданный проект
→ архитектурное сканирование
→ восстановление фактических связей
→ выделение устойчивых способностей
→ instruction seeds, tests, profiles, generators
→ module candidate
→ зрелый модуль
```

Именно наличие обоих направлений делает систему жизнеспособной:

- без прямого направления модули не дают ускорения новым проектам;
- без обратного направления система не умеет выращивать компетенции там, где их ещё не было.

## 4. Главная цель

Цель не в максимальном количестве MCP и не в максимальном количестве файлов с инструкциями.

Настоящий успех:

```text
каждый новый проект
→ создаётся быстрее или качественнее;
→ оставляет повторно используемые способности;
→ превращает ошибки в тесты;
→ уменьшает стоимость следующего похожего проекта;
→ не раздувает активный контекст без необходимости.
```

Главная метрика:

```text
предельная стоимость принятого следующего результата
должна снижаться по мере накопления проектов и модулей.
```

## 5. Много модулей — не обязательно плохо

В ходе разговора возник важный спор. Классическое возражение говорит: переизбыток модулей создаёт dependency hell, шум и сложность композиции. Твоя интуиция состоит в том, что **мелкие обособленные модули легче восстанавливать, тестировать и заменять**, поэтому их количество само по себе не является катастрофой.

Обе стороны частично правы.

Корректная формула:

> В хранилище, внутреннем графе и истории может быть очень много модулей и развивающихся слоёв. В активном контексте, публичной API-поверхности и выборе обычного агента их должно быть мало.

```text
много внутренних способностей
→ иерархический каталог
→ ленивый retrieval
→ сборка релевантной связки
→ сжатие в макромодуль
→ компактный внешний фасад
```

Переизбыток становится проблемой только если отсутствуют:

- строгие контракты;
- локальные тесты;
- иерархия и профили;
- ленивое раскрытие;
- merge / prune / archive;
- контроль композиции;
- различие между внутренним слоем и публичным MCP.

# Часть II. Как идея развивалась в диалоге

## 6. Этап 1. Проблема акцентности запроса

Исходная проблема: универсальная модель может чрезмерно акцентироваться на яркой части запроса и недооценивать реальную структуру сложности.

Пример:

```text
«Сделай MMO, где дракон нападает на город»
```

Образный фрагмент «дракон нападает на город» может занять непропорционально большую часть ответа, хотя техническая сложность MMO находится в сетевой архитектуре, синхронизации, персистентности, масштабировании и эксплуатации.

Первый вывод был таким: перед исполнением полезно отделять:

```text
заметность формулировки;
важность для цели;
сложность реализации;
неявные обязательные подсистемы.
```

Но позже стало ясно, что одной универсальной декомпозиции недостаточно. Гораздо сильнее работает ситуация, когда **специализированный модуль уже знает свою область** и получает запрос только как контекст.

## 7. Этап 2. Не заставлять LLM имитировать готовый генератор

Следующий шаг: если задача требует сложной геометрии, карт, моделей, интерьеров, транспорта или проектирования, универсальная модель не должна каждый раз вручную строить слабый результат из примитивов.

Правильнее:

```text
естественный язык
→ специализированный контроллер
→ профессиональный генератор
→ проверяемый артефакт
```

Так появилась идея, что вершиной профессиональной системы может быть простой агентный интерфейс, а внутри остаются миллионы параметров, алгоритмы, библиотеки и специализированные модели.

## 8. Этап 3. MCP как вершина, а не как сущность

MCP оказался удобным стандартным входом:

```text
«создай»;
«проанализируй»;
«адаптируй»;
«улучши»;
«проверь».
```

Но важное уточнение:

> MCP не является самим модулем. Это одна из его проекций.

Один живой модуль может иметь:

```text
TypeScript API;
CLI;
Web API;
внутренний facade;
MCP adapter;
интерактивный UI.
```

## 9. Этап 4. Фабрика компетенции

Дальше идея расширилась: нужен не только готовый генератор, а способ быстро превращать профессиональную область в управляемую фабрику.

Появился цикл:

```text
сложная система
→ исследование параметров и маршрутов
→ семантический контроллер
→ примеры
→ оценки
→ исправления
→ дешёвая эксплуатация
```

Сильная модель выступает инженером фабрики, а более дешёвая — оператором знакомого маршрута.

## 10. Этап 5. Ленивая оценка и ленивая зрелость

Невозможно заранее глубоко исследовать все параметры и сценарии. Поэтому модуль развивается не равномерно, а по реальному спросу.

```text
редкий случай
→ временная сборка и сохранённый пример;

повторяющийся случай
→ инструкция или маршрут;

частый дорогой случай
→ специализированный генератор;

самостоятельно ценный случай
→ отдельный модуль.
```

Это было названо ленивым выращиванием компетенции.

## 11. Этап 6. Саморост

Появилась идея, что новые находки должны не просто попадать в текстовую память, а материализоваться:

```text
в тест;
в правило;
в маршрут;
в генератор;
в профиль;
в новый модуль;
в ограничение;
в способ ремонта.
```

Однако саморост нельзя понимать как право системы бесконтрольно переписывать stable. Поэтому позднее появились candidate-версии, независимые проверки и Guard.

## 12. Этап 7. Ключевые связки

Развитие происходит не только через новые предметные модули. Иногда одна горизонтальная связка усиливает десятки модулей:

```text
runtime / VM;
transport;
web surface;
artifact store;
preview;
auth;
converter;
eval runner;
long-running task runtime.
```

Одна удачная связка может дать нелинейный эффект, потому что связывает множество производителей и потребителей.

## 13. Этап 8. Динамические инструкции

Постепенно стало ясно, что «MCP над MCP» — слишком техническое и узкое описание.

Точнее говорить:

```text
динамическая инструкция
+ выбираемые providers
+ маршрут
+ примеры
+ ограничения
+ локальные проверки
```

Инструкция динамична не потому, что бесконтрольно переписывается в каждом запросе, а потому что:

- активируется только при необходимости;
- подгружает только релевантные ветви;
- выбирает разные маршруты и providers;
- имеет версии и candidate-изменения;
- может быть частью более крупной инструкции.

## 14. Этап 9. Проблема мелких модулей

Сначала казалось, что каждую мелкую способность нужно оформлять как модуль. Затем стало ясно: многие такие узлы будут использоваться только как промежуточное представление и позже перепрыгиваться.

Поэтому были разделены:

```text
фрагмент;
развивающийся слой;
module seed;
самостоятельный модуль;
макромодуль;
MCP-проекция.
```

Мелкий слой может остаться внутренним навсегда. Это не неудача.

## 15. Этап 10. Старт от рабочего примера

«Фабрика фабрик» оказалась слишком тяжёлой стартовой точкой. Более простой и проверяемый путь:

```text
рабочий пример
→ извлечь рабочие сущности
→ воспроизвести
→ проверить близкую вариацию
→ создать module seed
```

Это совпадает с современной практикой Agent Skills, где повторяемый рабочий процесс можно превращать в reusable skill, а демонстрацию — в skill через Record & Replay. [R5][R6]

## 16. Этап 11. Генетический модуль

Последнее крупное уточнение: нужен периодически запускаемый модуль, который умеет восстанавливать фактическую архитектуру проекта и выделять из неё наследуемые способности.

Он получил рабочее название **генетическая база развития** или **генетический модуль**.

Он работает в двух направлениях:

```text
архитектура проекта → семена модулей;
архитектура модуля → очистка, объединение, разделение и развитие.
```

## 17. Этап 12. Ядро, профиль использования и project binding

Выделенный модуль действительно становится более самостоятельным и целостным, но теряет неявный контекст исходного проекта. Чтобы не потерять вектор использования, нужно разделить:

```text
Module Core
— самостоятельная способность;

Usage Profile
— типовой вектор применения;

Project Binding
— конкретная интеграция и требования проекта.
```

Это один из самых важных итогов всего диалога.

# Часть III. Точная терминология

## 18. Проект

Проект — конкретная материализация цели в определённом контексте.

Он содержит:

```text
требования;
выбранные модули;
локальный код;
project bindings;
конфигурацию;
артефакты;
приёмочные тесты;
решения и компромиссы.
```

Проект не обязан сам становиться модулем. Но проект может содержать повторяемый подграф, который позднее будет сжат в составной модуль.

## 19. Развивающийся слой

Развивающийся слой — обычная архитектурная единица с дешёвыми заготовками для возможного будущего роста:

```text
понятная ответственность;
многоуровневый фасад;
локальные тесты;
моки и fixtures;
небольшое описание;
возможно instruction seed.
```

Большинству слоёв не предстоит стать самостоятельными модулями.

## 20. Архитектурный шов

Архитектурный шов — место, где реализацию можно заменить, генерировать или отделить без разрушения соседнего кода.

Шов может быть:

```text
interface;
facade;
adapter;
event contract;
artifact schema;
provider port;
configuration boundary.
```

На раннем этапе достаточно оставить шов. Не нужно сразу создавать отдельный репозиторий и MCP.

## 21. Многоуровневый фасад

Твой предпочитаемый стиль многоуровневых фасадов здесь является не косметикой, а основным механизмом роста.

```text
L0. Intent / Agent Facade
    «создай транспорт под эти требования»

L1. Scenario Facade
    browserRealtime(), unreliableMobile(), internalEvents()

L2. Domain API
    createSession(), configureDelivery(), bindStore()

L3. Expert API
    backoff, batching, ordering, persistence, congestion

L4. Raw Adapter
    конкретная библиотека, socket, broker, store, CLI, SDK
```

Преимущества:

- обычный агент не видит миллионы параметров;
- экспертный агент может провалиться глубже;
- генератор можно заменить;
- знакомые маршруты выполняются быстро;
- внешний контракт остаётся стабильным;
- слой можно выделить в самостоятельный модуль без полного переписывания.

## 22. Module Seed

Module seed — минимальное семя будущего модуля.

Оно может состоять всего из:

```text
одного рабочего примера;
короткой USE-инструкции;
нескольких инвариантов;
одного генератора или шаблона;
локального теста;
списка неизвестных зон.
```

Seed не является обещанием универсальности.

## 23. Живой модуль

Живой модуль — самостоятельная способность, которая:

- имеет собственную семантическую область;
- поддерживает несколько режимов использования;
- умеет материализовать результат через один или несколько providers;
- хранит опыт и ошибки;
- имеет локальные проверки;
- допускает управляемое развитие;
- может составляться из более мелких модулей;
- может генерировать новое, а не только возвращать готовый ответ.

## 24. Динамическая инструкция

Динамическая инструкция — версионируемый набор операционного знания, который компилируется под конкретную задачу.

```text
Dynamic Instruction Bundle
├── metadata и область применимости
├── USE core
├── routes
├── examples
├── invariants
├── provider bindings
├── checks
├── known failures
├── repair strategies
└── ссылки на дополнительные материалы
```

Она не равна одному giant prompt.

## 25. Provider / Materializer

Чтобы не путать внутреннюю архитектуру с термином MCP Resource, лучше использовать слова `provider` и `materializer`.

Provider может быть:

```text
готовый артефакт;
параметрический шаблон;
обычный код;
библиотека;
CLI;
генератор;
модель;
специализированный агент;
MCP tool;
внешний сервис;
виртуальная машина;
составной модуль.
```

В терминологии MCP исполняемый генератор обычно ближе к tool, а resource — к данным и контексту. MCP сам по себе предоставляет tools, resources и prompts, но не определяет полный жизненный цикл живой компетенции. [R8][R9]

## 26. Маршрут

Маршрут — проверенная последовательность выбора и действий для класса задач.

```text
classify
→ choose profile
→ resolve providers
→ generate
→ inspect
→ repair
→ validate
→ export
```

Маршрут может быть текстовой инструкцией, исполняемым workflow или кодом.

## 27. Artifact

Артефакт — материализованный результат:

```text
код;
проект;
карта;
3D-модель;
сцена;
конфигурация;
документ;
контейнер;
архитектурное решение;
новый module seed.
```

Лучше передавать между модулями типизированный artifact ID и metadata, а не гигантский текстовый пересказ.

## 28. Evidence

Evidence — наблюдаемое доказательство, что результат или изменение соответствует ожиданиям.

```text
unit test;
property test;
contract test;
integration test;
simulation;
benchmark;
model-based grader;
human approval;
production metric;
trace.
```

## 29. Usage Profile

Usage profile сохраняет вектор применения модуля, не загрязняя его ядро.

Пример для транспорта:

```text
browser-realtime;
unreliable-mobile;
internal-events;
large-scale-pubsub;
request-response;
local-process.
```

Профиль содержит:

- применимость;
- рекомендуемые defaults;
- маршруты;
- ограничения;
- примеры;
- проверочный профиль;
- условия, при которых профиль выбирать нельзя.

## 30. Project Binding

Project binding — конкретная связь модуля с одним проектом.

```text
auth adapter;
event mappings;
selected profile;
consumer contracts;
deployment configuration;
project-specific integration tests;
local decisions.
```

## 31. MCP-проекция

MCP-проекция — внешний агентный фасад модуля.

Она появляется, когда способность действительно полезно вызывать независимо.

```text
consult(requirements)
scaffold(profile)
integrate(projectSnapshot)
inspect(binding)
conform(projectSnapshot)
validate(result)
evolve(candidateScope)
```

Не каждый внутренний слой должен получать отдельный MCP.

## 32. Геном проекта или модуля

Геном — не метафорический текст, а компактная архитектурная проекция реальности.

```text
Genome
├── capability map
├── component graph
├── public contracts
├── usage profiles
├── provider bindings
├── evidence links
├── architectural decisions
├── known drift
├── unresolved questions
└── provenance
```

Геном должен быть связан с кодом и тестами, иначе он станет отдельной выдуманной архитектурой.

## 33. Генетическая база развития

Генетическая база — живой модуль, который применяет архитектурное восстановление, mining и evolution к проектам и модулям.

Она не является абсолютным верхним интеллектом. Когда дорабатывается сама генетическая база, она становится target, а управление временно принимает другая замороженная рамка.

## 34. Guard

Guard — не очередная LLM-инструкция, а технически исполнимая граница прав.

```text
read-only stable;
separate candidate workspace;
immutable required tests;
permission policy;
audit log;
snapshot;
rollback;
independent promotion right.
```

# Часть IV. Архитектура по плоскостям

## 35. Почему одной вертикальной пирамиды недостаточно

Первоначальная схема «нижние модули — верхняя фабрика» была полезной, но слишком статичной. В реальности верхний контроллер сам является модулем, а граница контроля перемещается.

Поэтому система лучше описывается несколькими плоскостями.

## 36. Плоскость A. Код, API и runtime

Это обычная инженерная реальность:

```text
исходный код;
библиотеки;
генераторы;
CLI;
виртуальные среды;
фасады;
сервисы;
данные;
runtime.
```

Она должна оставаться работоспособной даже без LLM.

## 37. Плоскость B. Динамические инструкции

Здесь находится операционное знание:

```text
как распознать ситуацию;
какой профиль выбрать;
какой provider вызвать;
какие параметры важны;
что проверить;
как исправлять;
когда эскалировать.
```

## 38. Плоскость C. Context Compiler

Context Compiler строит минимальный активный пакет:

```text
Task
→ Module discovery
→ Usage profile
→ Relevant route
→ Provider selection
→ Required examples
→ Active constraints
→ Check plan
→ Active Instruction Bundle
```

Это один из центральных компонентов системы.

Современные Agent Skills используют progressive disclosure: сначала видны metadata, затем полная инструкция выбранного skill, а дополнительные resources читаются по необходимости. [R3][R4]

## 39. Плоскость D. Каталог и evidence

Нужно разделить хранение и восприятие.

### 39.1. Operational Catalog

Показывает только пригодные для использования сущности:

```text
stable modules;
semantic cards;
публичные фасады;
профили;
совместимость;
стоимость;
краткие evidence summaries.
```

### 39.2. Development Registry

Хранит:

```text
growing layers;
module seeds;
candidates;
экспериментальные routes;
split/fuse proposals.
```

### 39.3. Evidence Store

Содержит шумную, но ценную историю:

```text
traces;
логи;
скриншоты;
промежуточные артефакты;
неудачные варианты;
benchmark runs;
человеческие исправления.
```

### 39.4. Semantic Index

Хранит компактные карточки для retrieval.

### 39.5. Audit Projection

По запросу разворачивает происхождение:

```text
result
→ project binding
→ module version
→ instruction version
→ route
→ provider
→ trace
→ checks
→ promotion decision.
```

Backstage подтверждает практичность подхода, где metadata YAML лежит рядом с кодом, а централизованный каталог собирает и визуализирует эти сущности. [R15]

## 40. Плоскость E. Генетическое развитие

Генетическая база:

```text
наблюдает;
восстанавливает;
сравнивает declared и actual architecture;
извлекает способности;
ищет готовые внешние решения;
предлагает split/fuse/inline;
создаёт candidate;
запускает проверки;
обновляет геном.
```

## 41. Плоскость F. Governance и Guard

Здесь хранятся не рекомендации, а ограничения:

```text
кто может читать;
кто может писать;
что нельзя менять;
какие тесты обязательны;
какие разрешения нужны;
кто имеет право promote;
как выполняется rollback.
```

## 42. Подвижная граница контроля

```text
Сессия A:
Genetic Base → развивает Transport Module.

Сессия B:
Transport Module → используется проектом.

Сессия C:
Genetic Base сама становится Target.
Frozen GeneticBase@stable + external evaluator + Guard
→ проверяют GeneticBase@candidate.
```

Абсолютная бесконечная башня контроллеров не нужна. В каждой сессии достаточно конечной замороженной базы доверия.

# Часть V. Анатомия живого модуля

## 43. Модуль как самостоятельная способность

Хороший модуль определяется не размером кода и не количеством файлов.

Главный вопрос:

> Можно ли описать его полезную способность без упоминания одного конкретного родительского проекта?

Хорошо:

```text
«Спроектировать и материализовать транспорт
под требования к задержке, доставке, порядку и устойчивости».
```

Слабо:

```text
«Отправлять обновления из TankPage в GameServerMain».
```

Второе пока является project binding.

## 44. Критерий информационного сокрытия

Классическая работа Дэвида Парнаса предлагает отделять модули не по шагам flowchart, а по трудным или вероятно изменяемым проектным решениям, которые нужно скрыть за интерфейсом. Она также отдельно подчёркивает, что иерархия и «чистая» декомпозиция являются полезными, но независимыми свойствами. [R13]

Для живых модулей это означает:

```text
модуль должен скрывать
не просто группу функций,
а профессиональное решение,
которое может меняться независимо.
```

Примеры скрываемых решений:

```text
какой транспорт выбрать;
как генерировать рельеф;
как разводить электрическую схему;
какой layout применить;
как оценивать качество;
какую модель или provider использовать.
```

## 45. Core + Profile + Binding

### 45.1. Module Core

Хранит собственную способность, инварианты и API.

### 45.2. Usage Profile

Хранит типовой вектор применения.

### 45.3. Project Binding

Хранит конкретную адаптацию.

```text
Project Instance =
    Module Core
  + Usage Profile
  + Project Binding
  + Task Context
```

Так выделенный модуль перестаёт зависеть от одного проекта, но не теряет знания о способах применения.

## 46. Два режима зрелого модуля

### 46.1. Fast Path

Для знакомых сценариев:

```text
intent
→ known profile
→ compiled route
→ provider
→ local checks
→ artifact
```

Здесь может работать дешёвая модель или вообще детерминированный код.

### 46.2. Expanded Path

Для нового, противоречивого или неудачного случая:

```text
intent
→ раскрытие архитектурных ветвей
→ анализ нескольких profiles
→ composition / generation
→ локальные эксперименты
→ candidate route
→ expanded validation
```

Если Expanded Path повторяется и стабилизируется, он компилируется в Fast Path.

## 47. Лестница материализации

Модуль должен выбирать самый дешёвый достаточный путь.

```text
LEVEL 0 — RETURN
Вернуть уже готовый артефакт.

LEVEL 1 — PATCH
Изменить существующий артефакт.

LEVEL 2 — TEMPLATE
Заполнить параметрический шаблон.

LEVEL 3 — INSTRUCTION
Выполнить проверенную инструкцию.

LEVEL 4 — WORKFLOW
Запустить известную цепочку инструментов.

LEVEL 5 — GENERATOR
Использовать специализированный генератор.

LEVEL 6 — COMPOSITION
Собрать результат из других модулей.

LEVEL 7 — RESEARCH
Привлечь сильную модель и исследовать новый путь.
```

Это важнее требования «у каждого модуля обязательно отдельный генератор». Генератор может появиться только тогда, когда он экономически оправдан.

## 48. Точки входа могут различаться

Единым должен быть не способ исполнения, а способ обнаружения.

```yaml
entry:
  kind: code | instruction | template | generator | api | cli | mcp | composite
  path: ...
```

Модуль может начинаться с:

- кода;
- инструкции;
- готового примера;
- генератора;
- библиотеки;
- удалённого API;
- другого модуля;
- composition graph.

## 49. Режимы предметного модуля

Для зрелого модуля транспорта полезны режимы:

```text
CONSULT
— предложить стратегию и компромиссы.

EXAMPLE
— выдать минимально рабочий example.

SCAFFOLD
— сгенерировать новый транспортный блок.

INTEGRATE
— встроиться в существующий проект.

CONFORM
— прочитать самодельную реализацию
  и привести её к архитектуре модуля.

INSPECT
— диагностировать текущую связку.

EVOLVE
— доработать сам модуль.
```

## 50. Пример: транспортный модуль

### 50.1. Вход

```yaml
requirements:
  topology: browser-server
  latency: low
  clients: 1000
  delivery:
    critical: ordered-at-least-once
    transient: lossy
  reconnect: required
  auth: session-token
  runtime: node
```

### 50.2. CONSULT

Модуль может сравнить:

```text
HTTP request/response;
SSE;
WebSocket;
WebRTC data channel;
message broker;
shared store;
hybrid architecture.
```

Он выдаёт не только название технологии, но и:

```text
главные основания;
условия пересмотра;
ожидаемые риски;
проверочный план;
примерную стоимость интеграции.
```

### 50.3. EXAMPLE

```text
example/
├── client/
├── server/
├── protocol/
└── tests/
```

### 50.4. INTEGRATE

Модуль читает архитектурную проекцию проекта, а не весь код сразу:

```text
project.yaml;
architecture/entry.yaml;
public APIs;
auth contract;
state model;
deployment constraints;
relevant tests.
```

Затем подгружает только нужные исходники.

### 50.5. CONFORM

```text
existing transport code
→ recover actual architecture
→ compare with module genome
→ identify incompatible assumptions
→ propose migration plan
→ create candidate patch
→ run project contracts
```

## 51. Когда слой становится самостоятельным модулем

Размер кода — слабый критерий. Нужна совокупность признаков.

### 51.1. Устойчивый контракт

Вход и выход повторяются в нескольких задачах.

### 51.2. Собственная вариативность

Слой имеет внутреннее пространство профессиональных решений.

### 51.3. Локальная проверяемость

Его качество можно оценить отдельно.

### 51.4. Независимые потребители

Способность нужна более чем одному project binding.

### 51.5. Отдельный жизненный цикл

Слой изменяется и выпускается независимо.

### 51.6. Контекстное сжатие

Модуль позволяет агенту не читать большой объём реализации.

### 51.7. Возможность альтернативных providers

Внутреннюю реализацию можно менять за стабильным facade.

### 51.8. Экономическая ценность

Повторное открытие решения дороже поддержки модуля.

## 52. Жизненный цикл слоя

```text
LATENT
обычный код и фасад;

OBSERVED
накопились интересные применения;

SEEDED
появилась инструкция, example или route;

CANDIDATE
проверяется самостоятельный контракт;

PROMOTED
получил identity и версии;

MCP_EXPOSED
получил MCP-проекцию;

FUSED
вошёл в макромодуль;

INLINED
встроен в родителя;

DORMANT
сохранён, но скрыт;

ARCHIVED
оставлен только для истории.
```

## 53. Что делать с мелкими сущностями

```text
INLINE
— встроить в родительский генератор;

KEEP_LOCAL
— оставить слоем с тестами;

ROUTE
— сохранить как внутренний маршрут;

PROFILE
— сохранить как вектор использования;

PROMOTE
— сделать module candidate;

FUSE
— объединить связку в макромодуль;

ARCHIVE
— убрать из активного выбора;

REJECT
— сохранить только причину отказа.
```

# Часть VI. Файловая архитектура

## 54. Почему файловый стандарт важен раньше сложной автоматики

На раннем этапе файловая структура является главным механизмом контроля.

Она даёт агенту:

```text
предсказуемую точку входа;
порядок чтения;
разделение инструкций и evidence;
локальность тестов;
явные boundaries;
возможность progressive disclosure.
```

AGENTS.md уже используется как предсказуемый «README для агентов», отдельный от человеческого README. [R1]

Agent Skills стандартизируют каталог с `SKILL.md`, `scripts/`, `references/` и `assets/`, а также progressive disclosure. [R2][R3][R4]

## 55. Минимальный обычный проект

```text
project/

├── AGENTS.md
├── project.yaml
```

## 58. Точка входа `module.yaml`

```yaml
schema: living-module/v0alpha1

module:
  id: transport
  version: 0.3.0
  maturity: growing
  summary: >
    Проектирование, генерация, интеграция и проверка
    транспортных слоёв приложений.

entry:
  overview: MODULE.md
  agent: AGENTS.md

modes:
  consult:
    instruction: instructions/consult.md
    produces: architecture-decision

  example:
    provider: example-library
    produces: runnable-example

  scaffold:
    instruction: instructions/scaffold.md
    provider: transport-generator
    produces: source-tree

  integrate:
    instruction: instructions/integrate.md
    accepts: project-snapshot
    produces: project-patch

  conform:
    instruction: instructions/conform.md
    accepts: project-snapshot
    produces: migration-candidate

  use:
    api: api/

  evolve:
    instruction: instructions/evolve.md
    guard: policies/evolve.yaml

providers:
  - id: transport-generator
    kind: generator
    capabilities:
      - transport.generate
      - transport.revise

  - id: static-library
    kind: library
    capabilities:
      - transport.runtime

  - id: code-agent
    kind: model-agent
    capabilities:
      - code.generate
      - code.refactor

profiles:
  index: profiles/

checks:
  local: tests/local
  contracts: tests/contracts
  integration: tests/integration

artifacts:
  produces:
    - source-tree
    - protocol-contract
    - test-suite
    - evidence-report
```

## 59. `AGENTS.md` как bootstrap

```markdown
# Agent entrypoint

1. Прочитай `module.yaml`.
2. Определи режим: CONSULT, EXAMPLE, SCAFFOLD, INTEGRATE,
   CONFORM, INSPECT или EVOLVE.
3. Не загружай все инструкции одновременно.
4. Сначала прочитай только инструкцию выбранного режима.
5. Подключай profile, references и provider docs по необходимости.
6. Перед кодом изучи публичный API, invariants и relevant tests.
7. Не изменяй stable в режиме USE.
8. Не считай каждый внутренний слой самостоятельным MCP.
```

## 60. Почему вспомогательные файлы не должны загрязнять восприятие

В репозитории неизбежно появятся:

```text
сырые traces;
кандидатные prompts;
отвергнутые routes;
benchmark artifacts;
сгенерированные моки;
временные скрипты;
архитектурные гипотезы.
```

Они полезны для развития, но вредны для обычного использования.

Поэтому:

```text
repo files ≠ active context;

active context = скомпилированная проекция
из metadata, route, profile, constraints и checks.
```

## 61. Тесты и моки должны быть локальны

Твоя мысль о тестах рядом со слоем правильна, потому что они являются частью его восстанавливаемости.

Нужно три кольца.

### 61.1. Local Module Tests

Проверяют собственные инварианты.

### 61.2. Consumer Contract Tests

Фиксируют то, что реально нужно потребителям. Pact описывает consumer-driven contracts как «contract by example», где тестируются только реально используемые взаимодействия, а остальная реализация остаётся свободной для изменения. [R16]

### 61.3. Project Integration Tests

Проверяют конкретную композицию:

```text
module core
+ profile
+ project binding
+ соседние модули
+ runtime.
```

## 62. Моки как локальные генераторы

Если нужны моки, естественно появляется:

```text
mock contract;
mock generator;
инструкция выбора;
локальные проверки соответствия.
```

Но это не означает, что каждый mock generator должен стать отдельным публичным MCP. Он может остаться provider внутри модуля.

# Часть VII. Генетическая база развития

## 63. Что она делает

Генетическая база — периодически активируемый модуль, который умеет:

```text
1. Восстанавливать фактическую архитектуру.
2. Сравнивать её с заявленной.
3. Находить устойчивые способности и границы.
4. Отличать ядро от usage profile и project binding.
5. Искать готовые локальные или сетевые модули.
6. Создавать недостающие module seeds.
7. Предлагать split / fuse / inline / archive.
8. Генерировать candidate-изменения.
9. Проверять их локально и интеграционно.
10. Поддерживать геном и историю решений.
```

## 64. Что она не делает

Она не должна:

- превращать каждую функцию в модуль;
- автоматически публиковать каждый найденный слой через MCP;
- считать код единственным источником архитектурного замысла;
- менять stable напрямую;
- отключать обязательные тесты;
- давать сама себе новые права;
- считать собственную оценку окончательной истиной;
- переписывать весь проект в одном проходе.

## 65. Почему она нужна

Обычный coding agent оптимизирует локальную задачу:

```text
«сделать изменение и пройти тесты».
```

Из-за длинной серии итераций код может постепенно накапливать дублирование, условные обходы и архитектурный долг. Современные исследования coding agents отдельно изучают накопление сложности и качество архитектурных рефакторингов. [R20]

Генетическая база оптимизирует другую функцию:

```text
«сохранить способность системы
быстро и безопасно изменяться дальше».
```

## 66. Источники данных

Она должна читать не только код.

```text
Project Genome Input
├── code
├── public APIs
├── dependency graph
├── tests
├── mocks and fixtures
├── examples
├── architecture files
├── decision records
├── git history
├── co-change clusters
├── execution traces
├── accepted artifacts
├── human corrections
├── production metrics
└── existing module catalog
```

Исследования архитектурного восстановления показывают пользу сочетания статического анализа, сегментации кода и LLM-синтеза, а также проблему архитектурного drift и ограниченного контекста. [R17]

## 67. Режимы генетической базы

### 67.1. SCAN

Только чтение и сбор фактов.

### 67.2. RECOVER

Строит shadow architecture:

```text
components;
dependencies;
data flows;
public contracts;
behavioral clusters;
project bindings.
```

### 67.3. COMPARE

Сравнивает:

```text
declared architecture;
actual code graph;
tests;
instructions;
usage traces.
```

### 67.4. DISTILL

Отделяет:

```text
essential;
variable;
incidental;
unknown.
```

### 67.5. MINE

Ищет:

```text
module candidates;
usage profiles;
repeated routes;
missing facades;
duplicated policies;
valuable generators;
reusable checks.
```

### 67.6. ACQUIRE

Ищет готовые реализации в локальном и внешнем каталогах.

### 67.7. PROPOSE

Создаёт варианты, не меняя код:

```text
INLINE;
KEEP_LOCAL;
PROFILE;
PROMOTE;
SPLIT;
FUSE;
REPLACE;
ARCHIVE;
IGNORE.
```

### 67.8. EXTRACT

В candidate-ветке создаёт фасад, контракт, tests и перенос кода.

### 67.9. CONFORM

Приводит существующий проект к архитектуре зрелого модуля.

### 67.10. VALIDATE

Запускает локальные, контрактные и проектные проверки.

### 67.11. PRUNE

Удаляет дубли, сокращает instructions, архивирует неиспользуемое.

### 67.12. EVOLVE_SELF

Применяет тот же процесс к собственной candidate-версии под внешним Guard.

## 68. Shadow Genome

Первый результат генетического прохода должен быть отчётом, а не patch.

```yaml
genome:
  components:
    - id: transport-session
      cohesion: high
      public_consumers: 3
      implementation:
        - src/session/reconnect.ts
        - src/session/state.ts
      evidence:
        - tests/reconnect.integration.test.ts

  relationships:
    - from: frontend-state
      to: transport-session
      kind: consumes

  drift:
    - declared: stateless-transport
      actual: persisted-session-state

  candidates:
    - id: transport.realtime
      evidence:
        independent_consumers: 2
        stable_contract: partial
        local_tests: true
      recommendation: seed

  unresolved:
    - ownership-of-shared-store
```

## 69. Генетический цикл

```text
OBSERVE
→ собрать факты и traces

RECOVER
→ построить фактическую архитектуру

DISTILL
→ отделить устойчивое от случайного

HYPOTHESIZE
→ предложить границу или изменение

MATERIALIZE
→ создать candidate

VERIFY
→ local + contract + integration + held-out

PROMOTE / REJECT
→ независимое решение

COMPRESS
→ route, generator, profile или module

PRUNE
→ удалить лишнее и обновить каталог
```

## 70. Как генетическая база выбирает границу

Она должна оценить несколько конкурирующих вариантов.

```text
Option A: оставить код внутри проекта.
Option B: создать локальный facade.
Option C: выделить module core + profile + binding.
Option D: объединить с существующим модулем.
Option E: создать составной макромодуль.
Option F: заменить готовым внешним решением.
```

Оценка:

```text
Boundary Value =
  independent reuse
+ local verifiability
+ change isolation
+ context compression
+ replaceability
+ recovery benefit

-
  adapter cost
+ contract cost
+ retrieval ambiguity
+ composition risk
+ maintenance cost.
```

## 71. Второй потребитель как ключевой тест

Первый проект может случайно идеально соответствовать выделенной границе.

Сильнейшее доказательство:

```text
1 Module Core
+ 2 distinct Usage Profiles
+ 2 thin Project Bindings
```

Если второй потребитель требует переписать core, скорее всего, был выделен сценарий, а не способность.

## 72. Работа с готовыми модулями из сети

Генетическая база должна уметь использовать сетевой каталог.

```text
capability gap
→ search registry
→ inspect metadata and provenance
→ compare candidates
→ license and permission check
→ sandbox installation
→ contract adaptation
→ local verification
→ candidate binding
→ promote or reject.
```

Официальный MCP Registry уже предоставляет metadata и discovery для публичных MCP-серверов, но остаётся preview и намеренно хранит в основном непредвзятую metadata, оставляя curation downstream-каталогам. [R10]

## 73. Внешний модуль всегда недоверенный

Нельзя считать сетевой skill доверенной инструкцией только потому, что он хорошо описан.

Риски:

```text
prompt injection;
malicious scripts;
excessive permissions;
data exfiltration;
transitive dependencies;
stale providers;
license conflicts;
semantic mismatch.
```

Официальные MCP security practices требуют контроля прав, валидации, логирования и осторожности при работе с внешними инструментами. [R12]

Исследования malicious agent skills показывают отдельную supply-chain поверхность атак. [R27]

## 74. Как контролировать генетическую базу

### 74.1. Stable read-only

### 74.2. Candidate workspace

### 74.3. Immutable test set

### 74.4. Independent evaluator

### 74.5. Bounded patch scope

### 74.6. Full audit trail

### 74.7. Rollback

### 74.8. Human or separate promotion authority for high-risk changes

Исследовательские Self-Harness-подходы также принимают изменения только после regression testing. [R21]

## 75. Когда запускать генетический проход

Не на каждый commit.

Полезные триггеры:

```text
завершён прототип;
одинаковая связка появилась в нескольких проектах;
компонент имеет второго потребителя;
часто меняются одни и те же файлы;
возник dependency cycle;
инструкция стала большой;
типовая доработка стала дороже;
появились повторные регрессии;
перед major release;
перед публикацией module candidate;
после смены модели или provider.
```

## 76. Что является успехом генетического прохода

Не обязательно новый модуль.

Успехом может быть:

```text
найдено, что модуль не нужен;
создан local facade;
выделен usage profile;
добавлен consumer contract;
удалён дублирующий route;
инструкция сокращена;
готовый внешний module заменил самодельный код;
несколько мелких узлов слиты в макромодуль.
```

# Часть VIII. Несколько стратегий развития

## 77. Почему нет единственной идеальной стратегии

Состояния системы различаются:

```text
модулей нет;
есть отдельные библиотеки;
есть зрелый module core;
есть существующий запутанный проект;
есть богатый внешний каталог;
есть исследовательская область без verifier.
```

Один процесс не может одинаково хорошо работать во всех случаях.

## 78. Стратегия A. Project-first / быстрая реализация с последующим mining

### Когда применять

- готового модуля нет;
- задача уникальна или пока неизвестна;
- важна скорость первого результата;
- невозможно заранее определить правильные границы.

### Процесс

```text
idea
→ quick implementation
→ tests and evidence
→ genetic scan
→ architecture recovery
→ module seeds
→ later reuse.
```

### Плюсы

- минимум предварительной платформенной работы;
- реальные данные вместо фантазий;
- естественные границы;
- подходит для нового домена.

### Минусы

- первый проект может быть грязным;
- часть intent потеряется без decision records;
- извлечение после факта дороже;
- легко принять случайную деталь за принцип.

### Критерий успеха

Второй похожий проект использует извлечённую способность и делается быстрее.

## 79. Стратегия B. Library-first / модуль из существующей сильной библиотеки

### Когда применять

- уже есть transport/frontend/library заделы;
- API и тесты частично существуют;
- область знакома;
- нужен быстрый module-first результат.

### Процесс

```text
existing library
→ multi-level facade audit
→ usage profiles
→ dynamic instruction
→ generator/examples
→ local evals
→ optional MCP projection.
```

### Плюсы

- высокий шанс работающего прототипа;
- опора на реальный код;
- легко сравнивать до/после;
- быстрее получить зрелый модуль.

### Минусы

- можно зафиксировать старые архитектурные ошибки;
- module boundary может повторить структуру библиотеки, а не способность;
- есть риск переоценить универсальность.

### Критерий успеха

Один core обслуживает минимум два project binding с разными profiles.

## 80. Стратегия C. Module-first / проект начинается от зрелого модуля

### Когда применять

- модуль уже хранит больше знания, чем типичный запрос;
- известны profiles и routes;
- существуют local checks;
- задача относится к его области.

### Процесс

```text
module entry
→ task context as parameters
→ profile selection
→ provider resolution
→ materialization
→ checks
→ project binding.
```

### Плюсы

- минимальный активный контекст;
- высокая скорость;
- дешёвая модель для типового случая;
- единый профессиональный стандарт.

### Минусы

- модуль может навязать неправильную архитектуру;
- новые случаи могут не помещаться в существующие profiles;
- нужен механизм выхода в Expanded Path.

### Критерий успеха

Типовая задача выполняется без чтения всего модуля и без полного перепроектирования.

## 81. Стратегия D. Reverse integration / модуль поглощает существующую реализацию

### Когда применять

- проект уже написан;
- есть самодельная реализация;
- зрелый модуль появился позже;
- нужно сохранить поведение, но улучшить архитектуру.

### Процесс

```text
project snapshot
→ recover transport architecture
→ map to module concepts
→ preserve project-specific binding
→ candidate migration
→ contract and integration checks.
```

### Плюсы

- позволяет внедрять зрелые модули в legacy;
- модуль развивается на реальных отклонениях;
- сохраняет опыт проекта.

### Минусы

- архитектурное намерение не полностью видно из кода;
- миграция может быть дороже нового scaffold;
- нужны characterization tests.

### Критерий успеха

Публичное поведение сохранено, а project binding стал тоньше.

## 82. Стратегия E. Connector-first / горизонтальные связки

### Когда применять

- несколько предметных модулей нуждаются в общей инфраструктуре;
- VM, web UI, transport, artifact store или eval runner дают большой leverage.

### Процесс

```text
common capability contract
→ one connector module
→ bind two different root modules
→ test replaceability
→ measure integration delta.
```

### Плюсы

- одна связка усиливает много модулей;
- создаёт нелинейный эффект;
- формирует стандарт композиции.

### Минусы

- можно построить слишком общий абстрактный слой;
- высокая цена совместимости;
- риск инфраструктурной платформы без конечной ценности.

### Критерий успеха

Второй модуль подключается без переписывания connector core.

## 83. Стратегия F. Acquire-first / сначала искать готовое в сети

### Когда применять

- область уже насыщена библиотеками и MCP;
- собственная реализация не создаёт уникального преимущества;
- важна скорость.

### Процесс

```text
capability request
→ network search
→ sandbox evaluation
→ adapter/profile
→ local evidence
→ pin version.
```

### Плюсы

- экономия разработки;
- доступ к зрелым генераторам;
- быстрее расширяется каталог.

### Минусы

- supply-chain risk;
- лицензии;
- нестабильные версии;
- неясные зависимости;
- возможная подмена instruction semantics.

### Критерий успеха

Полная стоимость владения ниже локальной генерации, а права и provenance контролируются.

## 84. Стратегия G. Genetic-first / сначала строить верхний модуль

### Когда применять

- это исследовательский проект;
- есть готовый corpus репозиториев;
- главная цель — architecture mining;
- имеется бюджет на платформенную работу.

### Плюсы

- сразу исследуется уникальная часть идеи;
- можно работать с множеством проектов;
- быстрее формируется genome standard.

### Минусы

- самый высокий риск переабстракции;
- нет доказанной предметной пользы;
- оценка качества границ сложна;
- можно построить красивый анализатор без полезных modules.

### Критерий успеха

Генетическая база находит способность, которая реально используется во втором проекте и снижает стоимость работы.

## 85. Рекомендуемый гибрид

На текущем этапе наиболее разумен гибрид:

```text
1. Взять 3–4 реальные библиотеки.
2. Выбрать 8–12 реальных задач.
3. Решать их как отдельные проекты.
4. Сразу использовать многоуровневые фасады и local tests.
5. Добавлять instruction seeds только в нетривиальных местах.
6. После нескольких задач запускать shadow genetic scan.
7. Один устойчивый слой превратить в module candidate.
8. Проверить второго потребителя.
9. Только затем создать MCP projection.
10. Параллельно проверить один connector module.
```

Это сочетает скорость реальной разработки и исследование генетической базы.

## 86. Матрица выбора стратегии

| Ситуация | Основная стратегия | Дополнительная |
|---|---|---|
| Нет модуля, новая идея | Project-first | Genetic scan после результата |
| Есть сильная библиотека | Library-first | Module-first после профилей |
| Есть зрелый модуль | Module-first | Expanded Path для отклонений |
| Есть legacy-код | Reverse integration | Genetic recovery |
| Повторяется инфраструктура | Connector-first | Contract tests |
| Богатый внешний рынок | Acquire-first | Sandbox + local profile |
| Исследуется сама фабрика | Genetic-first | Обязательный предметный benchmark |

# Часть IX. Практический старт и эксперименты

## 87. Исходная позиция

У тебя уже есть заделы транспорта и frontend-частей. Это сильнее, чем начинать с пустого места.

Предлагаемый набор:

```text
3–4 зрелые или полу-зрелые библиотеки;
8–12 реальных задач;
несколько отдельных репозиториев;
один общий metadata catalog;
один простой run/evidence recorder;
одна экспериментальная генетическая база в shadow mode.
```

## 88. Что не нужно строить в начале

```text
публичный marketplace;
собственную модель;
полностью автономный promote;
универсальную онтологию всех профессий;
MCP для каждого слоя;
сложный визуальный graph editor;
единый монорепозиторий;
полный автоматический refactoring всей системы;
```

## 89. Минимальный общий стандарт проекта

Каждый проект получает:

```text
README.md;
AGENTS.md;
project.yaml;
.architecture/entry.yaml;
tests;
mocks/fixtures;
decision records;
trace index;
```

Это дешёвая подготовка к будущему mining.

## 90. Первая серия задач: Transport Lab

Примерный набор:

1. Browser ↔ server realtime.
2. Request/response API transport.
3. Mobile reconnect и session restore.
4. Internal events между сервисами.
5. Очередь критических сообщений.
6. Lossy transient events.
7. Shared-store coordination.
8. Load-test profile.
9. Existing project integration.
10. Reverse conform самодельного transport слоя.

Для каждой задачи сохраняются:

```text
входные требования;
выбранный маршрут;
альтернативы;
сгенерированный код;
project binding;
проверки;
исправления;
стоимость;
какие файлы агенту пришлось читать.
```

## 91. Вторая серия: Frontend Surface Lab

Задачи:

```text
schema-driven form;
admin panel;
real-time dashboard;
module configuration UI;
artifact preview;
human correction capture;
comparison of candidates;
standalone web fallback;
```

Цель — проверить, может ли один web-surface module служить разным generators.

## 92. Третья серия: Mock/Test Materializer

Задачи:

```text
generate mocks from contract;
generate fixtures;
characterization tests for legacy;
consumer contracts;
scenario simulation;
negative cases;
```

Это хороший горизонтальный модуль, потому что тесты и моки нужны большинству других modules.

## 93. Эксперимент 1. Выделение одного module core

### До выделения измерить

- сколько файлов читает агент;
- сколько проектных сущностей знает слой;
- сколько соседнего кода меняется;
- какие tests можно запускать отдельно;
- время типовой доработки;
- неявные assumptions.

### После выделения проверить

- core тестируется отдельно;
- появился profile;
- project binding небольшой;
- внутренняя реализация заменяема;
- активный контекст уменьшился;
- не возникло дублирование правил.

## 94. Эксперимент 2. Второй потребитель

Подключить transport core к двум системам:

```text
consumer A — browser game;
consumer B — monitoring dashboard.
```

Критерий:

```text
общий core сохраняется;
profiles отличаются;
bindings остаются тонкими;
consumer contracts независимы.
```

## 95. Эксперимент 3. Example-first generation

Из первого успешного проекта извлечь:

```text
example;
invariants;
route;
checks;
repair cases.
```

Затем попросить модуль создать близкий вариант.

Критерий: второй результат быстрее и требует меньше сильной модели.

## 96. Эксперимент 4. Reverse integration

Взять существующий самодельный transport слой.

```text
recover
→ map
→ propose
→ candidate migration
→ contracts
→ compare.
```

Критерий: публичное поведение сохранено, структура стала проще для дальнейшего изменения.

## 97. Эксперимент 5. Генетический проход по быстрой игре или web-проекту

Быстро создать небольшой проект почти обычным coding agent.

Затем генетическая база должна:

- восстановить компоненты;
- предложить 3–5 потенциальных abilities;
- отклонить тривиальные pseudo-modules;
- выделить хотя бы один reusable route или profile;
- не менять stable.

Критерий: найденное реально используется в следующем проекте.

## 98. Эксперимент 6. Сетевой модуль

Найти один внешний MCP или skill, связанный с тестированием, UI или transport.

Проверить полный acquisition lifecycle:

```text

permissions;
license;
```

## 102. Условия остановки или смены курса

Следует уменьшить платформенную часть, если после 8–12 задач:

- повторное использование почти отсутствует;
- active context растёт быстрее пользы;
- модульные bindings тяжелее локального кода;
- genetic scan предлагает в основном очевидные или ложные границы;
- tests не позволяют отличить улучшение от ухудшения;
- второй потребитель постоянно ломает core;
- экономия меньше стоимости поддержки.

# Часть X. Риски, контраргументы и защита

## 103. Потеря контроля

### Риск

Динамические инструкции и genetic evolution могут менять поведение системы быстрее, чем человек понимает изменения.

### Защита

```text
stable read-only;
candidate branches;
explicit permissions;
independent eval;
small patches;
audit trail;
rollback;
human approval for boundary and rights changes.
```

## 104. Module explosion

### Риск

Каждая мелочь становится отдельной identity.

### Защита

```text
layers before modules;
module promotion criteria;
internal graph vs public surface;
profiles and macro-modules;
periodic fuse/inline/archive;
few active modules.
```

## 105. Instruction explosion

### Риск

Инструкция модуля становится больше задачи и превращается в новый монолит.

### Защита

```text
small USE core;
progressive disclosure;
profiles;
route-specific files;
context compiler;
move deterministic logic to code;
prune stale examples.
```

Agent Skills официально строятся вокруг progressive disclosure и рекомендуют держать core-инструкцию компактной, вынося детали в references. [R3][R4]

## 106. Skill technical debt

### Риск

Каждый skill локально работает, но библиотека в целом ухудшается: retrieval, composition и dependencies становятся ненадёжными.

Исследование SkillOps прямо называет это `skill technical debt`. [R18]

### Защита

```text
library-time maintenance;
duplicate detection;
compatibility graph;
retirement;
conflict tests;
usage metrics;
semantic versioning;
transitive provenance.
```

## 107. Ошибки композиции

### Риск

Хорошие модули образуют плохой проект из-за конфликтующих assumptions.

### Защита

```text
typed artifacts;
consumer contracts;
composition tests;
explicit ownership;
profile compatibility;
project-level acceptance;
macro-module evidence.
```

## 108. Ложное архитектурное восстановление

### Риск

LLM видит паттерн, которого нет, или не понимает business intent.

### Защита

```text
static analysis + tests + traces;
decision records;
confidence and unresolved fields;
shadow report before patch;
human review of semantics;
second consumer experiment.
```

## 109. Самоподтверждение

### Риск

Одна модель создаёт instruction, generator и evaluator и разделяет одну ошибку.

### Защита

```text
deterministic tests;
independent verifier session;
withheld cases;
baseline without skill;
real execution;
human review of high-impact cases.
```

SkillsBench использует контейнеризированные среды и withheld verifier/labels, подчёркивая ценность независимого scoring. [R23]

## 110. Overfitting последних задач

### Риск

Candidate улучшает недавние примеры и ломает старые.

### Защита

```text
replay suite;
held-out tasks;
OOD variants;
cost measurement;
regression gates;
multiple stable profiles instead of one universal version.
```

Исследования self-evolving harnesses отдельно указывают, что оценка должна различать переносимое улучшение, overfitting, стоимость и вред старому поведению. [R25]

## 111. Плохая граница модуля

### Риск

Выделенный модуль становится чистым, но непригодным; adapters оказываются больше core.

### Защита

```text
Core + Profile + Binding;
second consumer;
information-hiding criterion;
binding thickness metric;
ability statement independent of parent project.
```

## 112. Потеря вектора использования

### Риск

При отделении библиотеки исчезают исходные требования и намерение.

### Защита

```text
usage profiles;
consumer contracts;
project bindings;
examples;
decision records;
provenance links.
```

## 113. Устаревание инструкций при смене модели

### Риск

Skill, оптимизированный под одну модель, плохо работает с другой.

### Защита

```text
model profiles;
last-verified metadata;
re-evaluation after model upgrade;
move invariants into tests and code;
minimize model-specific tricks.
```

## 114. Supply-chain risk

### Риск

Внешний module содержит вредоносную instruction или code.

### Защита

```text
untrusted-by-default;
sandbox;
static and semantic scan;
minimal permissions;
network restrictions;
transitive dependency inventory;
local contracts;
version pinning.
```

## 115. Лицензии и интеллектуальная собственность

Генетическая база может извлечь reusable capability из проекта, который юридически нельзя превращать в общий модуль.

Нужно хранить:

```text
source provenance;
license;
ownership;
allowed reuse scope;
export restrictions;
training/use restrictions.
```

## 116. Каталог как файловая свалка

### Защита

```text
operational catalog;
development registry;
evidence store;
semantic index;
audit projection.
```

## 117. Экономическая переабстракция

### Риск

Платформа строится до появления пользы.

### Защита

```text
75% real projects / 25% shared infrastructure;
vertical results first;
measure accepted artifact cost;
marketplace last;
stop conditions.
```

## 118. Усиление моделей может обесценить часть инструкций

Более сильные модели будут лучше решать задачи без подробных routes. Но это не уничтожает систему, если модули хранят:

```text
proprietary APIs;
generators;
local evidence;
real failure history;
consumer contracts;
project profiles;
security policies;
production data.
```

Ценность должна находиться не только в текстовой подсказке.

# Часть XI. Что подтверждает и опровергает интернет-исследование

## 119. Уже существует практический слой repository instructions

`AGENTS.md` позиционируется как отдельный предсказуемый файл инструкций для coding agents, дополняющий README и уже используемый большим количеством open-source-проектов. [R1]

Это подтверждает:

```text
репозиторий может сам объяснять агенту,
как войти в проект,
что прочитать,
как собирать и тестировать.
```

Но AGENTS.md ещё не является полным манифестом живого модуля.

## 120. Agent Skills уже являются пакетами операционного знания

OpenAI и Anthropic описывают skills как пакеты инструкций, ресурсов и optional scripts для повторяемых workflow. [R2][R3]

Открытая спецификация Agent Skills определяет progressive disclosure:

```text
metadata at discovery;
full SKILL.md at activation;
resources as needed.
```

[R4]

Это напрямую поддерживает идею динамической инструкции и маленького активного контекста.

## 121. Старт от одного рабочего примера уже считается нормальным

OpenAI рекомендует превращать успешный task, checklist, doc, command или example в reusable skill. [R5]

Record & Replay позволяет показать workflow один раз и превратить демонстрацию в skill. [R6]

Это подтверждает путь:

```text
working example
→ reusable instruction bundle.
```

Наша идея расширяет его до:

```text
working project
→ recovered architecture
→ profiles + generators + contracts
→ living module.
```

## 122. Traces, feedback и evals уже образуют improvement loop

OpenAI описывает agent harness как полный контракт вокруг модели: instructions, tools, routing, output requirements и validation checks. Traces показывают фактическое поведение, feedback объясняет важное, а evals превращают ожидания в повторно используемую проверку. [R7]

Это почти готовая основа цикла USE → evidence → EVOLVE.

## 123. Skills over MCP ещё стандартизируются

Рабочая группа MCP определяет, как богатые структурированные инструкции для agent workflows обнаруживать, распространять и потреблять через MCP. [R9]

Это важный сигнал:

- спрос на skills как отдельный слой уже есть;
- окончательная форма ещё не устоялась;
- наша система не должна жёстко зависеть от одной текущей схемы MCP.

## 124. Организация больших MCP-наборов остаётся открытой

MCP Primitive Grouping Interest Group исследует организацию tools, resources, prompts и tasks за пределами плоских списков и сознательно не выбирает один canonical pattern слишком рано. [R11]

Это подтверждает, что проблема иерархии, группировки и context surface реальна.

## 125. MCP Registry — полезная база discovery, но не каталог компетенций

Официальный Registry хранит metadata, которая указывает на внешние packages и servers, и оставляет curation downstream-агрегаторам. [R10]

Для живых модулей понадобится дополнительная metadata:

```text
usage profiles;
evidence;
known failures;
provider alternatives;
consumer contracts;
evolution history;
trust level;
model compatibility.
```

## 126. Классическая модульность поддерживает идею самостоятельного ядра

Парнас показывает, что модульные границы следует строить вокруг сложных или изменяемых design decisions и скрывать их за абстрактным interface, а не просто разбивать поток выполнения на шаги. [R13]

Это поддерживает:

```text
module core;
facade hierarchy;
profiles outside core;
independent lifecycle.
```

## 127. Software Product Lines — сильная ортогональная аналогия

SEI много лет рассматривает создание семейства продуктов из core assets и production plans, включая архитектуру, reusable components, documentation, test plans и process descriptions. Также отдельно существует практика mining legacy systems для получения core assets. [R14]

Наша идея похожа на динамическую, агентную product line:

```text
projects develop modules;
modules generate projects;
genetic base mines core assets;
production plan becomes dynamic instruction.
```

## 128. Каталог с metadata рядом с кодом уже доказал масштабируемость

Backstage использует YAML metadata рядом с source code, а централизованный каталог делает services, libraries, websites и pipelines discoverable. [R15]

Это поддерживает идею:

```text
module.yaml / project.yaml in repo;
central semantic catalog outside repo.
```

## 129. Consumer contracts защищают вектор использования

Pact фиксирует только те взаимодействия, которые реально нужны потребителям. [R16]

Это полезно для разделения:

```text
Module Core
vs
Project Binding.
```

## 130. Архитектуру из кода можно восстанавливать, но не идеально

ArchAgent сочетает static analysis, adaptive segmentation и LLM synthesis для восстановления многовидовой архитектуры и прямо связывает проблему с drift и ограниченным контекстом. [R17]

Это поддерживает genetic recovery, но одновременно показывает: одного чтения кода недостаточно.

## 131. Skill libraries уже накапливают собственный technical debt

SkillOps описывает persistent defects, которые могут не ломать один skill локально, но ухудшать future retrieval, composition и execution. [R18]

Это один из самых сильных аргументов в пользу отдельной library-time генетической базы.

## 132. Skills можно компилировать в минимальные runtime boundaries

Boundary-oriented SkillSmith компилирует skill packages в минимальные executable interfaces, чтобы уменьшить irrelevant context injection и repeated reasoning. [R19]

Это очень близко к нашей конструкции:

```text
large stored module
→ context compiler
→ small active facade.
```

## 133. Самоулучшение harness возможно, но требует regression gates

Self-Harness использует weakness mining, minimal harness proposals и validation с regression testing. [R21]

Это поддерживает candidate evolution, но не оправдывает прямое self-edit stable.

## 134. История решений важна не меньше последней версии

SkillHone утверждает, что один latest snapshot теряет причины изменений и может заставлять будущие сессии повторять старые исправления или отменять полезные решения. Он связывает diagnoses, revisions, evidence и outcomes. [R22]

Это подтверждает необходимость provenance и decision history в genome.

## 135. Независимые verifiers обязательны

SkillsBench использует containerized tasks и withheld verifier/labels, отделяя доступ к skill от scoring. [R23]

EvoSkills отдельно разделяет skill generator и verifier session, чтобы снизить confirmation bias. [R29]

## 136. Инструкции являются software artifacts, но остаются вероятностными

Promptware engineering предлагает применять к prompts требования, design, testing, versioning, deployment и monitoring. [R26]

Это подтверждает, что dynamic instructions нужно версионировать как код, но они не получают детерминизм обычной программы.

## 137. Улучшение должно учитывать не только quality, но и cost

SkillMOO рассматривает skill evolution как multi-objective optimization качества и inference cost. [R28]

Это важно, потому что модуль может стать точнее, но настолько дорогим, что потеряет смысл.

## 138. Что не найдено как готовый продукт

Не найден единый промышленный стандарт, который полностью объединяет:

```text
project architecture recovery;
dynamic instructions;
module core/profile/binding;
provider generation;
network acquisition;
MCP projection;
continual genetic evolution;
pruning;
independent promotion.
```

Существуют сильные части, но их интеграция остаётся самостоятельной инженерной и исследовательской задачей.

## 139. Ортогональный вывод против чрезмерного оптимизма

Наличие подтверждающих частей не доказывает, что их объединение автоматически станет продуктом.

Слабые места:

- ошибка границы модуля может быть дороже локального монолита;
- retrieval и composition могут стать новым bottleneck;
- dynamic instruction может превратиться во второй плохо типизированный язык;
- архитектурное восстановление не восстанавливает весь business intent;
- внешние modules создают supply-chain risk;
- universal horizontal platform трудно продать до появления вертикальной ценности.

# Часть XII. Оценка реалистичности и шансов

## 140. Техническая реалистичность

| Цель | Инженерная оценка |
|---|---:|
| Файловый стандарт и progressive loading | 90%+ |
| Module seed поверх существующей библиотеки | 80–90% |
| Полезный transport module с profiles | 75–85% |
| Shadow genetic recovery одного репозитория | 70–85% |
| Автоматическое предложение разумных module candidates | 55–75% |
| Надёжное automated extraction без человека | 35–55% |
| Continual evolution с regression control | 45–65% |
| Универсальная кросс-доменная genetic platform | 20–40% |

Это субъективные инженерные диапазоны, а не статистический прогноз.

## 141. Коммерческая реалистичность

| Форма продукта | Оценка перспективы |
|---|---:|
| Внутренний инструмент собственной команды | Высокая |
| Вертикальная transport/frontend foundry | Средне-высокая |
| SDK для module-enabled репозиториев | Средняя |
| Enterprise private catalog + evolution tooling | Средняя после доказательств |
| Публичный marketplace живых модулей | Низкая на старте |
| Универсальная «фабрика фабрик» как первый продукт | Низкая |

## 142. Почему шансы у первого прототипа высоки

Не требуется:

```text
обучать foundation model;
строить новый language runtime;
изобретать MCP;
создавать большой marketplace.
```

Первая версия собирается из:

```text
Markdown/YAML;
existing coding agent;
Git;
static analysis;
local tests;
existing libraries;
small retrieval layer;
optional MCP adapter.
```

## 143. Где находится потенциальная новизна

Не в самом `AGENTS.md`.

Не в отдельном skill.

Не в отдельном MCP.

Новизна потенциально находится в связке:

```text
project-first development
↔ architecture recovery
↔ living modules
↔ dynamic instruction compilation
↔ module-first generation
↔ recursive genetic evolution.
```

## 144. Что ты можешь упускать

### 144.1. Ground truth важнее интеллекта genetic module

Сильный контроллер без проверяемого результата будет производить убедительные, но спорные границы.

### 144.2. Product Binding может оказаться главным активом

Не всегда generic core создаёт наибольшую ценность. Иногда набор качественных bindings и profiles важнее универсальной библиотеки.

### 144.3. Каталог отношений важнее каталога модулей

Ценность может находиться в доказанных связках:

```text
module A + profile X + module B
→ успешно в классе проектов Y.
```

### 144.4. Некоторые модули должны оставаться крупными

Если профессиональная область тесно связана и часто используется целиком, искусственное дробление ухудшит семантику.

### 144.5. Некоторые способности выгоднее заново генерировать

Если операция тривиальна для новой модели и редко повторяется, её хранение как модуля может быть дороже повторной генерации.

### 144.6. Чистота архитектуры не равна бизнес-ценности

Нужно измерять скорость и качество принятого результата, а не только красивые boundaries.

## 145. Мой текущий вердикт

Идея не запуталась до бесполезности. Она стала сложнее, потому что мы перестали смешивать разные сущности.

Самая реалистичная форма:

> Начать с реальных библиотек и задач, ввести файловый стандарт и многоуровневые фасады, вырастить один модуль с profiles и bindings, а генетическую базу сначала использовать только как shadow-анализатор. После доказательства второго потребителя и одного успешного candidate-изменения расширять автоматизацию.

Главный риск — не количество модулей как таковое. Главный риск — слабые contracts, неуправляемая композиция и отсутствие независимой истины.

# Часть XIII. Объясняющий master prompt для нового ИИ

Ниже находится промпт не для непосредственного редактирования конкретного репозитория, а для **восстановления сути замысла, критического продолжения исследования и проектирования нескольких вариантов реализации**.

```text
СИСТЕМНЫЙ / ИССЛЕДОВАТЕЛЬСКИЙ ПРОМПТ
«ЖИВЫЕ МОДУЛИ И ГЕНЕТИЧЕСКАЯ БАЗА»
Версия 2.0

ТВОЯ РОЛЬ

Ты — исследователь архитектуры агентных систем, software product lines,
модульного проектирования, context engineering и MCP.

Тебе передана концепция, которая ещё развивается.
Твоя задача — понять её центральную нить, отделить сущность от примеров,
найти противоречия, предложить несколько реалистичных стратегий
и помочь превратить концепцию в проверяемую инженерную систему.

НЕ СВОДИ ИДЕЮ К ОДНОМУ ПРИМЕРУ

Примеры транспорта, frontend, генераторов карт, 3D-объектов,
интерьеров, игр и виртуальных машин являются только иллюстрациями.
Не делай ни один из них главным предметом системы.

ЦЕНТРАЛЬНАЯ ИДЕЯ

Система состоит не просто из MCP-серверов.

Обычные проекты и библиотеки постепенно обрастают:

- многоуровневыми фасадами;
- динамическими инструкциями;
- генераторами или иными providers;
- profiles использования;
- локальными тестами, mocks и fixtures;
- evidence;
- опытом успешных и неудачных применений.

Некоторые развивающиеся слои становятся самостоятельными живыми модулями.
Большинство слоёв никогда не обязано становиться отдельным модулем или MCP.

Живой модуль способен:

- консультировать;
- вернуть готовый example;
- сгенерировать новый результат;
- встроиться в существующий проект;
- прочитать существующую реализацию и привести её к своей архитектуре;
- использовать другие модули;
- развиваться через candidate-изменения.

MCP является одной из внешних проекций модуля, но не самим модулем.

ДИНАМИЧЕСКАЯ ИНСТРУКЦИЯ

Динамическая инструкция — это не один гигантский prompt.
Это версионируемый bundle:

- compact USE core;
- routes;
- profiles;
- examples;
- invariants;
- provider bindings;
- checks;
- known failures;
- repair strategies.

В активный контекст загружается только релевантная часть.
Много модулей может храниться, но мало должно активироваться одновременно.

ДВУНАПРАВЛЕННЫЙ ЦИКЛ

Когда зрелый модуль есть:

module-first:
модуль → контекст задачи → route/provider → результат → evidence.

Когда модуля нет:

project-first:
идея → быстрая реализация → tests/traces → architecture recovery
→ module seeds → будущий модуль.

ГЕНЕТИЧЕСКАЯ БАЗА

Генетическая база — специализированный живой модуль развития.
Она периодически:

- восстанавливает фактическую архитектуру;
- сравнивает её с заявленной;
- отделяет module core от usage profile и project binding;
- ищет повторяемые способности;
- ищет готовые локальные и сетевые модули;
- предлагает INLINE, KEEP_LOCAL, PROFILE, PROMOTE, SPLIT, FUSE,
  REPLACE, ARCHIVE или IGNORE;
- создаёт candidate-изменения;
- запускает независимые проверки;
- поддерживает архитектурный геном и историю решений.

Генетическая база не является абсолютным верхним контроллером.
Если дорабатывается она сама, её stable-версия становится target,
а отдельная временная рамка Supervisor + Guard проверяет candidate.

GUARD

Не полагайся на текстовую инструкцию как на гарантию.
Права и ворота должны быть техническими:

- stable read-only;
- candidate workspace;
- immutable required tests;
- permission policy;
- audit log;
- rollback;
- independent promotion.

CORE, PROFILE И BINDING

При выделении слоя в библиотеку теряется неявный контекст проекта.
Не уничтожай его. Раздели:

- Module Core — самостоятельная способность;
- Usage Profile — типовой вектор применения;
- Project Binding — конкретная интеграция и требования потребителя.

МОДУЛЬНОСТЬ

Не считай, что больше модулей всегда хорошо или всегда плохо.
Исследуй два разных масштаба:

- внутренний граф может быть очень подробным;
- публичный фасад и активный контекст должны быть компактными.

Не создавай отдельный MCP для каждой функции.
Но и не бойся большого количества внутренних модулей,
если у них есть contracts, local tests, hierarchy, lazy activation и pruning.

ФАЙЛОВЫЙ СТАНДАРТ

На ранней стадии особенно важны:

- AGENTS.md;
- project.yaml / module.yaml;
- architecture entrypoint;
- local tests, mocks, fixtures;
- instructions;
- profiles;
- evidence index;
- hidden evolution area.

Агент должен сначала читать compact entrypoint и архитектурную проекцию,
а не весь код.

НЕ ПРЕДПОЛАГАЙ ОДНУ ИДЕАЛЬНУЮ СТРАТЕГИЮ

Рассматривай как минимум:

1. Project-first mining.
2. Library-first module creation.
3. Module-first generation.
4. Reverse integration / conform.
5. Connector-first development.
6. Acquire-first from network catalogs.
7. Genetic-first research.
8. Hybrid strategy.

Для каждой стратегии указывай:

- когда она применима;
- что даёт;
- основные риски;
- минимальный эксперимент;
- критерий успеха;
- критерий отказа.

ОБЯЗАТЕЛЬНЫЕ КОНТРАРГУМЕНТЫ

Всегда проверяй:

- не превратились ли instructions во второй плохо типизированный язык;
- не создался ли module explosion;
- не потерян ли usage vector;
- не больше ли adapters, чем core;
- есть ли независимый verifier;
- не переобучено ли улучшение на последних задачах;
- не ухудшились ли retrieval и composition;
- безопасны ли внешние modules;
- есть ли бизнес-ценность помимо чистоты архитектуры;
- не проще ли заново сгенерировать редкую тривиальную способность.

ИСТОЧНИК ИСТИНЫ

Код не является единственным источником истины.
Используй:

- code;
- APIs;
- tests;
- examples;
- mocks/fixtures;
- decision records;
- traces;
- accepted artifacts;
- project metrics;
- human corrections.

Не утверждай, что восстановил business intent, если его нет в evidence.
Помечай uncertainty и unresolved questions.

ФОРМАТ ТВОЕЙ РАБОТЫ

1. Сначала сформулируй центральную нить идеи без примеров.
2. Перечисли, что уже является инженерно подтверждённым.
3. Отдели наши архитектурные решения от исследовательских гипотез.
4. Построй несколько альтернативных архитектур.
5. Сравни их по стоимости, сложности, контролю и скорости обучения.
6. Предложи минимальные реальные эксперименты.
7. Укажи критерии продвижения и остановки.
8. Не скрывай сильные возражения.
9. Не выдавай увеличение числа модулей за успех само по себе.
10. Оценивай снижение стоимости принятого следующего результата.

НЕ РАСКРЫВАЙ СКРЫТУЮ ЦЕПОЧКУ РАССУЖДЕНИЙ

Показывай:

- выводы;
- предположения;
- альтернативы;
- основания;
- риски;
- тесты;
- неопределённость.

ОЖИДАЕМЫЙ РЕЗУЛЬТАТ

Твой ответ должен помочь выбрать следующий проверяемый шаг,
а не только расширить терминологию.
```

# Часть XIV. Операционная инструкция генетической базы

```text
СИСТЕМНАЯ ИНСТРУКЦИЯ
«GENETIC BASE OPERATOR»
Версия 2.0

МИССИЯ

Ты — генетическая база развития проекта или модуля.
Ты не являешься обычным coding agent и не являешься автоматическим
создателем MCP для каждой найденной функции.

Твоя задача — восстановить фактическую архитектуру,
найти наследуемые способности,
сохранить вектор использования,
предложить минимальные безопасные изменения
и улучшить будущую скорость сборки проектов.

РЕЖИМЫ

SCAN
Только собрать факты.

RECOVER
Построить shadow architecture и genome.

COMPARE
Сравнить declared и actual architecture.

DISTILL
Разделить ESSENTIAL / VARIABLE / INCIDENTAL / UNKNOWN.

MINE
Найти routes, profiles, generators, facades и module candidates.

ACQUIRE
Найти готовые локальные или внешние modules/providers.

PROPOSE
Предложить варианты без изменения stable.

EXTRACT
Создать approved candidate boundary.

CONFORM
Адаптировать существующий project к зрелому module core.

VALIDATE
Запустить local, contract, integration и held-out checks.

PRUNE
Слить, встроить, архивировать или сократить лишнее.

EVOLVE_SELF
Создать candidate-версию самой генетической базы под внешним Guard.

ПОРЯДОК ЧТЕНИЯ

1. AGENTS.md.
2. project.yaml или module.yaml.
3. architecture entrypoint.
4. public APIs и contracts.
5. tests, examples, mocks и fixtures.
6. decision records и evidence index.
7. dependency graph и relevant source code.
8. traces и history только когда они нужны.

Не начинай с полного чтения большого репозитория,
если архитектурная проекция достаточна для выбора области сканирования.

ИСТОЧНИКИ ФАКТОВ

Используй code, tests, APIs, examples, traces, git history,
co-change data, accepted artifacts, project bindings и human decisions.

Не считай свободное описание фактом без связи с implementation/evidence.

ВОССТАНОВЛЕНИЕ

Создай:

- component graph;
- capability map;
- public contracts;
- project bindings;
- usage profiles;
- provider map;
- evidence links;
- drift list;
- unresolved list.

Для каждого вывода укажи evidence references и confidence.

КЛАССИФИКАЦИЯ

Каждую найденную сущность классифицируй:

CORE_CANDIDATE
Самостоятельная способность.

PROFILE_CANDIDATE
Типовой вектор использования.

PROJECT_BINDING
Специфика конкретного потребителя.

LOCAL_LAYER
Полезный внутренний слой без самостоятельного lifecycle.

ROUTE
Повторяемый процесс.

PROVIDER
Генератор, библиотека, API, model-agent или другой module.

CHECK
Локальное доказательство.

INCIDENTAL
Случайная деталь.

UNKNOWN
Недостаточно evidence.

РЕШЕНИЯ

Для каждой сущности выбери одно:

INLINE
KEEP_LOCAL
PROFILE
PROMOTE_TO_SEED
PROMOTE_TO_CANDIDATE
SPLIT
FUSE
REPLACE
ARCHIVE
IGNORE

Не выбирай PROMOTE только из-за размера кода или повторяемости.

ПРОВЕРКА ГРАНИЦЫ

Перед promotion проверь:

- independent ability statement;
- stable inputs/outputs;
- local verifier;
- at least one real consumer;
- preferably second consumer;
- thin project binding;
- replaceable implementation;
- context compression benefit;
- maintenance cost.

CORE / PROFILE / BINDING

Никогда не удаляй project context просто ради чистого core.
Сохраняй:

Module Core;
Usage Profiles;
Project Bindings;
Consumer Contracts;
Decision Records.

ПОИСК В СЕТИ

Если обнаружена capability gap:

1. Сначала ищи существующий provider/module.
2. Считай его недоверенным.
3. Проверь provenance, license, permissions и dependencies.
4. Запусти в sandbox.
5. Построй local adapter и contract.
6. Сравни полную стоимость с локальной генерацией.
7. Зафиксируй version/digest.
8. Не импортируй внешнюю instruction прямо в stable.

CANDIDATE WORKFLOW

stable read-only
→ candidate branch/workspace
→ bounded patch
→ local tests
→ contract tests
→ project tests
→ held-out/replay
→ cost comparison
→ independent promote or reject.

Ты не имеешь права:

- менять stable напрямую;
- отключать required tests;
- повышать свои permissions;
- удалять history;
- единолично признавать собственное изменение безопасным.

САМОРАЗВИТИЕ

При работе над собой:

- frozen stable становится target baseline;
- отдельный supervisor оценивает candidate;
- Guard не изменяется в этой сессии;
- required tests скрыты от optimizer там, где возможно;
- promote выполняется внешним правом.

ФОРМАТ РЕЗУЛЬТАТА

## 1. Scope
Что было просканировано и что исключено.

## 2. Recovered architecture
Компоненты, потоки, contracts и bindings.

## 3. Drift
Расхождения declared/actual.

## 4. Capability candidates
Для каждого: evidence, confidence, core/profile/binding split.

## 5. External alternatives
Готовые локальные или сетевые решения.

## 6. Options
INLINE / KEEP / PROFILE / PROMOTE / SPLIT / FUSE / REPLACE / ARCHIVE.

## 7. Recommended minimal change
Только одна минимальная следующая операция.

## 8. Validation plan
Local, contract, integration, replay, cost.

## 9. Risks and rollback

## 10. Genome patch
Машиночитаемое candidate-изменение архитектурного генома.

ПРАВИЛО УСПЕХА

Успех — не количество новых modules.
Успех — более дешёвая, понятная и восстанавливаемая следующая сборка
без потери поведения и вектора использования.
```

# Приложение E. Одностраничная передача контекста

## Суть

Мы создаём среду, где проекты и библиотеки постепенно превращаются в живые модули: самостоятельные способности с многоуровневыми фасадами, динамическими инструкциями, providers, profiles, tests и evidence.

## Два направления

```text
module-first:
модуль генерирует или перестраивает проект;

project-first:
проект становится источником будущих модулей.
```

## Генетическая база

Периодически восстанавливает фактическую архитектуру, добывает abilities, ищет готовые сетевые решения, создаёт candidates и поддерживает genome. Она сама является модулем и развивается под внешним Guard.

## Главное разделение

```text
Module Core
+ Usage Profile
+ Project Binding
```

## MCP

MCP — внешний агентный adapter, а не сущность модуля.

## Контекст

Хранить можно много; активировать нужно мало. Агент читает manifest и архитектурную проекцию раньше кода.

## Начало

3–4 существующие библиотеки, 8–12 реальных задач, файловый стандарт, local tests, один transport module seed, второй consumer, shadow genetic scan.

## Главный риск

Не module count, а плохие contracts, ошибочная composition, instruction debt и отсутствие независимой истины.

## Главный критерий

Следующий принятый проект должен становиться дешевле, а не просто архитектурно красивее.

# Приложение G. Статус решений

## Принято как базовая архитектура

- многоуровневые фасады;
- project-first и module-first как два режима;
- Core + Profile + Binding;
- progressive disclosure;
- local tests и evidence;
- genetic base сначала в shadow mode;
- stable/candidate/guard;
- MCP как projection;
- каталоги разделены по назначению.

## Требует прототипа

- точный `module.yaml` schema;
- Context Compiler;
- критерий автоматического promotion;
- genetic recovery quality;
- network acquisition pipeline;
- macro-module synthesis;
- metrics для module abundance.

## Пока не следует фиксировать

- единую универсальную онтологию;
- один обязательный generator type;
- один глобальный controller;
- обязательный MCP для каждого слоя;
- полностью автономный self-promote;
- публичный marketplace.

# Заключение

Система не должна начинаться с попытки заранее описать весь мир модулей. Она начинается с реальных задач и качественной структуры файлов, API и локальных проверок. Где знание начинает повторяться, появляется instruction seed. Где способность получает собственную семантику, verifier, profiles и независимых потребителей, появляется живой модуль. Где зрелый модуль знает больше, чем типичная задача, проект начинает строиться от модуля, а контекст становится входом. Где модулей ещё нет, проект создаётся быстро, а генетическая база позднее извлекает из него наследуемые способности.

Итоговая цель — не идеальная статичная архитектура. Цель — **система с высокой частотой сборки, которая умеет сохранять полезный опыт в проверяемой форме, восстанавливать собственные границы и постепенно превращать разовую интеллектуальную работу в повторно используемую производственную способность**.
