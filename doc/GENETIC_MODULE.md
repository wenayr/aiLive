# Генетический модуль

## Определение

Генетический модуль — не память сама по себе и не автономный разум. На первом
этапе это маленькая ответственная программа, которая обрабатывает наблюдаемые
модули с учётом их зрелости: замечает изменение, выбирает применимую инструкцию,
запрашивает ограниченную оценку и обновляет состояние после решения владельца.

Рядом с программой хранятся отдельные данные:

- реестр наблюдений и наблюдаемых путей;
- активные, уточнённые и снятые инструкции;
- допустимые исключения вроде «здесь старый вызов намеренный»;
- история срабатываний и решений;
- счётчики полезных, ложных и пропущенных замечаний.

Поэтому модуль можно заменить или улучшить, не выдавая накопленный JSON за саму
логику. И наоборот, один и тот же алгоритм может работать с разной памятью.

## Минимальный цикл

1. При инициализации снимается SHA-256 baseline наблюдаемых файлов.
2. После завершённой правки агент явно запускает scan. Git для этого не нужен.
3. Если релевантной инструкции ещё нет, возникает одно действие `discover`.
4. Владелец либо записывает короткое наблюдение, watched paths и инструкцию,
   либо выбирает `skip`, если устойчивого правила пока нет.
5. Следующая связанная правка вызывает только `inspect` этой инструкции.
6. `clear` завершает цепочку; только `issue` возвращается владельцу для решения
   `keep`, `refine` или `retire`.
7. Всё состояние сохраняется и восстанавливается новым процессом.

Одновременно существует не более одного pending action. Новый scan запрещён,
пока предыдущее действие не разрешено. Это делает причинную цепочку читаемой.

## Выборочный вектор

Модуль не обязан проверять весь проект или скрывать сложность всегда. Он
показывает представление, соответствующее запросу. Большая реализация может
выглядеть как один «генератор данных» или как шесть ответственных модулей, если
для текущей задачи именно такие границы полезны. При отладке остаётся доступ к
подмодулям и исходному коду.

На нижнем уровне инструкция может быть почти механической:

- этот helper должен вызываться из перечисленных методов;
- у этой библиотеки есть типичная ошибка использования;
- старый стиль допустим только в явно записанных местах;
- при изменении адаптера проверь общий normalizer.

Дешёвая проверка рассматривает только этот вектор. Более сильная оценка нужна,
если локальная проверка нашла неоднозначность или нарушение.

## Созревание

Модуль растёт только вслед за реальным повторением:

1. обычный код без отдельной конструкции;
2. наблюдение и короткая инструкция;
3. детерминированный guard или локальная проверка;
4. группа связанных подмодулей с историей частых ошибок;
5. evaluator, mock generator или генератор повторяющегося кода;
6. самостоятельный MCP/API/facade, если граница уже доказала ценность;
7. позднее — генеративное и динамическое проектирование реализации.

Генератор появляется не потому, что «генераторы полезны», а когда есть реальное
повторение, понятный source of truth и автоматическая проверка результата. Это
может быть семейство биржевых API, 3D-модели, таблицы, моки или любой другой
повторяющийся ресурс.

## Критерий пользы

Главные метрики — повторная ошибка, число ремонтов, время, токены/стоимость,
ложные срабатывания и накладные расходы самого цикла. Рабочая гипотеза ранней
стадии: генетическая модульность не должна регулярно отнимать больше примерно
20% усилий, а её ценность должна проявляться на следующей похожей задаче.
Количество модулей, инструкций и логов само по себе успехом не является.
