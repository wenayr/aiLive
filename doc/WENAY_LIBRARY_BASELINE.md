Дата фиксации: 2026-07-15.

- `wenay-common2@1.0.76` — `C:\Users\wenay\AppData\Roaming\npm\node_modules\wenay-common2`;
- `wenay-react2@1.0.52` — `C:\Users\wenay\AppData\Roaming\npm\node_modules\wenay-react2`.

Глобальная установка — это общая справочная база агента. Она **не** добавляет
пакет в зависимости продукта и не даёт продукту права на неявный deep import.
Для конкретного проекта подключение остаётся отдельным архитектурным решением:
явная запись в `package.json`, воспроизводимая установка и импорт только из
публичного entrypoint.

## Обязательная стартовая проверка

Перед созданием новой повторяемой логики агент обязан:

1. Определить, является ли задача общей инфраструктурой или политикой конкретного продукта.
2. Прочитать публичные документы и exports обеих библиотек, относящиеся к этой
   задаче. Для `wenay-common2` это минимум `README.md`, `CLAUDE.md` и
   `doc/wenay-common2.md`; для `wenay-react2` — `README.md`,
   `doc/PROJECT_RULES.md`, `doc/PROJECT_FUNCTIONALITY.md`,
   `doc/EXAMPLE_USAGE.md` и `doc/wenay-react2.md`.
3. Выбрать существующий примитив, если он уже владеет нужным lifecycle,
   состоянием, подписками, persistence или UI-взаимодействием.
4. Если подходящего примитива нет — коротко зафиксировать, почему нельзя
   переиспользовать существующий, и только затем писать новую логику.

Проверка относится к shared-логике; она не требует искусственно протащить
React или common2 в маленькую автономную страницу, генератор или probe.

## Правила, извлечённые из библиотек

### `wenay-common2`

- Строить reusable units как фабрики-замыкания: явный `deps` на границе,
  состояние и внутренние helpers — внутри замыкания.
- Возвращаемый объект — осмысленный API. Разделять фасады (`control`, `api`,
  `debug`) только когда у них действительно разные потребители.
- Не смешивать ресурс/adapter, чистую utility-логику и локальную business
  policy. Утилиты извлекать, а policy оставлять над общим примитивом.
- Не менять публичный интерфейс и не дублировать существующую логику без
  отдельного решения.

### `wenay-react2`

- Выбирать самый высокий существующий примитив, который уже владеет нужным
  lifecycle. Новый shared behavior сначала проектировать как headless
  `use*` hook или `create*` controller; визуальный компонент должен быть
  тонким renderer-слоем.
- Держать policy продукта в product wrapper, а не в библиотечном примитиве.
- В потребителях импортировать из package root. Deep imports допустимы лишь
  при работе внутри самой библиотеки либо пока нет публичного export.
- Для shared UI предпочитать классы и CSS custom properties. Inline styles
  оставлять для вычисляемой runtime-геометрии; новые token добавлять только
  когда значение повторяется или должно переопределяться темой.
- Сохранение UI-состояния решает приложение; shared primitive может только
  сигнализировать о dirty state.

## Общий стиль кода

Во всём host-репозитории по умолчанию действуют: named functions для
нетривиальной логики, `==`/`!=` вместо строгих сравнений без необходимости,
одинарные кавычки, четыре пробела и отсутствие `;`. Публичные типы выводятся
из реализации там, где это не ухудшает читаемость.

## Актуализация

Перед значимым новым потоком проверять опубликанные версии (`npm view`) и
локально установленные manifests. Если версия изменилась, сначала читать
свежие `doc/changes/` пакетов, затем обновлять этот документ и только после
этого принимать новые архитектурные решения.
