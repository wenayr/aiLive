# Созревание модуля, MCP-проекция и генетический поиск

Дата: 2026-07-13.

Статус: исследовательская гипотеза из рабочего диалога.

## Исходная мысль

Генетический модуль должен периодически рассматривать код и существующие модули, трезво оценивать их текущее состояние и выращивать вокруг перспективных способностей более сильную форму управления.

На ранней стадии агент видит обычный код, читает API и самостоятельно решает, как им воспользоваться. Более зрелый модуль накапливает:

- краткое семантическое введение;
- стандартные сценарии применения;
- удачные примеры и контрпримеры;
- типичные ошибки;
- диагностику причин;
- способы исправления;
- локальные оценщики;
- маршруты создания, изменения и проверки результата.

Тогда агенту не нужно каждый раз заново восстанавливать профессиональный способ применения из исходников. Он получает небольшую задачу или пример, а модуль разворачивает проверенный процесс.

## Предлагаемая траектория

    обычный код
    → описанный слой
    → модуль с примерами и troubleshooting
    → модуль с evaluator и repair routes
    → операционный контроллер
    → самостоятельный runtime
    → MCP-проекция, если нужен агентный доступ

Важное уточнение: MCP сам по себе не содержит профессионального опыта. Он только предоставляет стандартизированную поверхность доступа. Зрелость находится в contracts, instructions, workflows, examples, evaluators, diagnostics и накопленном evidence.

Поэтому правильнее говорить не «модуль стал умным, потому что стал MCP», а:

> Модуль накопил операционную компетенцию и при необходимости получил MCP-проекцию.

## Самоприменение генетического модуля

Генетический модуль также может проходить собственную лестницу зрелости:

    периодически запускаемая инструкция
    → воспроизводимый SCAN workflow
    → registry наблюдений и candidates
    → evaluator registry
    → candidate manager и package foundry
    → оркестратор модулей
    → MCP-проекция

Самоприменение не должно означать самоподтверждение. Изменение генетического модуля требует замороженной предыдущей версии, независимого reviewer, holdout-задач и rollback.

## Что означает «генетический»

Генетичность системы шире буквального генетического алгоритма:

- наследование — удачные contracts, profiles, evaluators и routes переходят в новую версию;
- вариация — создаются альтернативные boundaries, generators, instructions и compositions;
- отбор — candidates сравниваются по внешнему evidence;
- сохранение — версии, lineage и rejected hypotheses не теряются;
- рекомбинация — несколько модулей или стратегий образуют новый composite;
- мутация — ограниченное изменение одного элемента;
- разнообразие — система не сворачивается преждевременно в один архитектурный стиль.

Случайность является только одним возможным оператором вариации. Она не должна быть источником смысла и не заменяет архитектурные гипотезы.

## Буквальный эволюционный режим

Позднее можно сравнивать поколения стратегий:

- быстрые и грубые SCAN-проходы;
- медленные глубокие проходы;
- разные decomposers;
- разные policies поиска generators;
- разные наборы examples;
- разные модели и бюджеты;
- разные composition routes.

Однако поколение имеет смысл только при фиксированном lineage, сопоставимых входах, независимом fitness и контроле стоимости. Иначе «поколения» будут просто несравнимыми запусками с разным random seed.

## Каталог текущих проектов

Для обучения на реальной практике нужен versioned project catalog. Минимально он хранит:

- project snapshot или commit;
- цели и non-goals;
- домен и consumers;
- acceptance tests;
- фактические layers и dependencies;
- usage traces;
- найденные module/generator candidates;
- принятые и отвергнутые изменения;
- внешние решения;
- стоимость и результаты повторных задач.

Каталог нужен не только для поиска готового модуля, но и как набор контрастных сред для проверки переноса.

## Гипотезы для проверки

1. Модуль с operational knowledge уменьшает ошибки и активный контекст по сравнению с тем же кодом без такой оболочки.
2. MCP-проекция полезна только после появления устойчивых семантических операций.
3. Структурированные мутации превосходят случайный поиск при малом числе дорогих экспериментов.
4. Независимый project catalog позволяет отличить повторно используемую способность от особенности одного проекта.
5. Генетический модуль может улучшать собственные workflows без деградации holdout-сценариев.
