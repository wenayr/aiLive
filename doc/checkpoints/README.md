# Checkpoints: сводка состояния лаборатории

Checkpoint — короткий, проверяемый снимок проекта. Он собирает подтверждённое,
неизвестное, противоречия, решения и следующий ход из разрозненных traces,
evaluations, plans и кода. Это не новый источник фактов и не замена evidence:
каждое существенное утверждение должно ссылаться на первичный артефакт.

## Зачем это нужно

Пробы, explorations и agent attempts намеренно создают много частичных следов.
Без регулярной сборки они выглядят как один неразличимый поток, а старые
неизвестные ошибочно принимаются за решённые. Checkpoint превращает этот поток
в читаемое состояние, не удаляя raw evidence.

`CURRENT.md` — входная точка для человека и следующего agent run. `INDEX.md`
хранит историю снимков. Старые checkpoint-файлы неизменяемы после публикации;
исправление факта оформляется следующим checkpoint с явной ссылкой на прежний.

## Когда создавать checkpoint

Создание event-driven: checkpoint нужен после любого из событий ниже.

- Завершён или остановлен заметный delivery slice, SCAN, experiment либо agent
  batch.
- Появилось evidence, меняющее решение, инвариант, риск или статус objective.
- Обнаружено противоречие, failed acceptance либо неясная ownership/recovery
  граница.
- Изменился режим проекта: например, BUILD → SCAN, PROFILE → transfer check.
- Перед передачей работы другому человеку/агенту или перед публикацией.
- Если событий долго нет — после пяти существенных traces/evaluations в одной
  кампании либо перед еженедельным обзором.

Не нужен checkpoint после косметической правки, повторного зелёного запуска без
нового вопроса или промежуточной мысли без evidence. Такие материалы остаются в
notes, trace или progress.

## Как собрать

1. Сначала перечислить первичные evidence и результаты проверок; не выводить
   выводы из памяти диалога.
2. Разнести каждый пункт по одному status: `confirmed`, `contradicted`,
   `unknown`, `blocked`, `in-progress` или `decision`.
3. Отделить наблюдение от интерпретации. Если проверка не была запущена, это
   `unknown`, даже если решение кажется очевидным.
4. Выбрать один ближайший проверяемый ход и явные non-goals до следующего
   checkpoint. Не превращать сводку в бесконечный backlog.
5. Добавить запись в `INDEX.md`, обновить `CURRENT.md`, а при изменении правила
   — решение в `doc/decisions/DECISIONS.md`.

## Обязательный шаблон

```md
# Checkpoint YYYY-MM-DD — краткое имя

Статус: active | paused | handed-off
Период: ...
Область: ...
Основание: ссылки на trace, evaluation, test или decision

## Подтверждено
## Противоречия и дрейф
## Неизвестно / заблокировано
## Решения с прошлого checkpoint
## Следующая ставка
## До следующего checkpoint не делаем
## Указатели на evidence
```

В формулировках используются даты и конкретные границы: «P1 не проверен», а не
«транспорт почти готов». Один checkpoint должен помещаться в один экран или
быть близок к этому; подробности остаются по ссылкам.

## Связь с agent ecology

Каждая завершённая пачка scout/builder/adjudicator работ создаёт checkpoint либо
явно ссылается на существующий. Agent consensus не может сам обновить
`confirmed`: этот статус требует deterministic check, независимого review или
другого указанного evidence. Дорогой adjudicator может собрать противоречия и
предложить следующий ход, но не отменяет это правило.
