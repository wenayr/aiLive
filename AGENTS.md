# AI Live — рабочие правила

## Порядок чтения

1. `project.yaml`.
2. `doc/lab/FOUNDATION.md` и `doc/lab/ARCHITECTURE.md`.
3. Релевантный `layer.yaml`, contracts и локальные tests.
4. Реализацию, history и большой исследовательский корпус — только по необходимости.

## Код и границы

- До создания новой логики проверь, нет ли уже подходящего primitive в `wenay-common2` или `wenay-react2`.
- Пиши через фабрики-замыкания, а не классы: экспортируемая фабрика принимает зависимости объектом; внутренние функции могут замыкаться на состояние.
- Возвращаемый объект фабрики — осмысленный facade. Делить его на `control`, `api`, `runtime`, `debug` или `testing` нужно по аудитории, а не механически.
- Держи `resource`, чистые `utility`, локальную business logic, координацию и bindings раздельно. Resource core не импортирует React, Socket.IO, файловый executor или consumer facade.
- Facade проецирует каноническую операцию и не копирует её реализацию. Debug-поверхность может быть шире runtime-поверхности, но не обходит policy.
- Новая переиспользуемая UI-логика начинается с headless `create*`/`use*` controller; JSX — тонкий слой поверх него. Продуктовая терминология остаётся в app wrapper, а не в shared primitive.
- Используй `==`/`!=`, одинарные кавычки, четыре пробела и отсутствие `;`. Для нетривиальных обработчиков выбирай именованные функции. Не аннотируй возвращаемый тип без необходимости.
- Primitive и union aliases получают префикс `t`; публичные типы предпочтительно выводятся через `ReturnType<typeof createX>`.

## Runtime и безопасность

- L0 — наблюдаемый local development runner, а не security sandbox. Не заявляй иную степень изоляции.
- UI никогда не передаёт shell-команду. Runner исполняет только зарегистрированные `task.json`, без shell interpolation и только в ограниченном task/workspace path.
- Не добавляй SSH, publish, package install, сетевую загрузку или произвольный terminal API в runtime facade без отдельного решения и явной команды пользователя.
- Живое состояние и логи находятся в `.laboratory/` и не коммитятся. В git попадают только curated evidence, спецификации и fixtures.

## Проверки и документация

- Для нетривиальной работы создай короткий временный файл в `doc/progress/`, обновляй его на границах этапов и удали после завершения.
- `README.md` — навигация. Долговечное описание идёт в `doc/lab/`, решения — в `doc/decisions/`, факты эксперимента — в `doc/evidence/`.
- Новая внешняя зависимость сначала требует отдельного `ACQUIRE`/quarantine решения. Не добавляй её «по пути».
- Базовая проверка: `npm run verify`. Если она не проходит, зафиксируй точную причину в evidence или progress, а не объявляй работу завершённой.
